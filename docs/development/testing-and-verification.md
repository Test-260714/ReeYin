# 测试与验证

## 必须做

- [ ] 按“取最高等级”原则判定 R0-R4，并把变更类型映射到本规范的专项验证矩阵。
- [ ] 从 L0 差异检查开始，按风险逐层推进；代码变更优先构建最小受影响项目，再扩展到测试、消费者和场景。
- [ ] 在最终修改后重新运行验证，保留精确命令、执行时间、环境、退出码、错误/失败/跳过计数和关键输出。
- [ ] 将命令证据与人工场景证据分开记录；工具确实不提供的字段写“`不适用`”，禁止猜测或补造。
- [ ] 区分本次失败、既有失败、模拟验证、真实设备验证和未验证项，不得扩大任何证据的解释范围。
- [ ] 覆盖正常、边界、失败、取消、超时、停止、恢复、重试、并发和资源释放路径中与变更相关的部分。
- [ ] 对公共契约、配置、Recipe、Output、序列化、缓存、数据库和模块加载变更构建消费者并验证兼容路径。
- [ ] R2 公共契约/兼容边界取得人类领域评审；R3 还须领域评审通过并取得变更负责人批准后才能交付或进入受控验证；R4 再取得覆盖具体设备或数据目标、操作、执行人、窗口、检查及停止/回滚/恢复的执行批准，且不得由 AI 执行。
- [ ] 未获得与风险匹配的新鲜证据时，不得声称“完成”“修复”“通过”或“可发布”。

本规范细化根级 [强制规则](../../AGENTS.md) 与 [统一贡献流程](../../CONTRIBUTING.md)。安全前置条件见 [安全与信息安全](safety-and-security.md)，评审和证据交付格式见 [评审与交付](review-and-delivery.md)，AI 执行验证时还必须遵守 [AI 开发规范](ai-development.md)。

## 风险等级与最低门禁

一个变更命中多个类别时采用最高等级；无法确定时按较高等级处理。下表在根级门禁概览基础上明确 R3/R4 的评审与批准语义：

| 等级 | 典型范围 | 最低门禁概览 |
| --- | --- | --- |
| R0 | 文档、注释且不改变运行行为 | 差异、内容、路径和链接检查 |
| R1 | 局部 UI、内部逻辑、非公共重构 | R0 + 修改项目构建 + 相关测试 |
| R2 | 公共 API、MVVM 生命周期、Recipe、Output、序列化、缓存、数据库 | R1 + 消费者构建 + 兼容性与场景测试；涉及公共契约/兼容边界时必须由熟悉该领域的人评审 |
| R3 | 硬件运动、报警、权限、安全、迁移、发布配置 | R2 + 失败/恢复路径 + 停止/回滚方案 + 领域评审通过 + 变更负责人批准 |
| R4 | 对设备、人身或生产数据存在不可逆风险 | R3 + 真实环境方案 + 具体 R4 执行批准，AI 不得执行 |

风险等级只规定最低门禁，不是验证上限。影响范围扩大、失败代价升高、证据不足或实施中出现新风险时，必须暂停并更新风险等级、设计、验证、评审与授权。R2 涉及公共契约或兼容边界时，必须取得熟悉该领域的人类评审结论。R3 必须覆盖失败/恢复路径和停止/回滚方案，由人类领域评审通过，并由明确的变更负责人批准后才能交付或进入受控验证。R4 除满足 R3 外，还必须有真实环境验证方案，并取得覆盖具体设备或数据目标、操作、执行人、窗口、检查及停止/回滚/恢复的 R4 执行批准；AI 不得执行任何设备、数据或其他对象的 R4 操作。

领域评审是技术、安全或兼容性的 sign-off 结论；变更批准是允许交付、合入或进入下一阶段。基础受控操作授权按对象分为真实设备操作授权、生产数据变更授权，以及任务明确要求的其他高风险对象授权；每项授权必须限定具体设备或数据目标、操作、执行人、窗口、检查及停止/回滚/恢复。风险分类为 R4 时，相关受控操作授权升级为 R4 执行批准并满足 R4 额外门禁。例外批准只允许在有效期内延期非安全验证，不能替代领域评审、变更批准、受控操作授权或 R4 执行批准。

R0 文档变更不要求代码构建，但必须执行 L0，并检查内容准确性、相对路径、内部/外部链接和是否存在失效或互相冲突的规则。文档若改变生成、配置、运行步骤或公共契约，则不能仅按 R0 处理。

## 验证层级

验证按低成本、低副作用到高成本、高风险的顺序执行。高层结果不能替代低层检查，低层结果也不能证明高层行为。

| 层级 | 名称 | 必须回答的问题 | 典型证据 |
| --- | --- | --- | --- |
| L0 | 差异 | 最终差异是否只包含任务范围，文件、内容、路径、链接和敏感信息是否正确 | 文件清单、差异审阅、链接解析结果 |
| L1 | 静态 | 语法、格式、分析器、警告策略、依赖方向和静态契约是否满足 | 格式/分析器/搜索命令及完整输出 |
| L2 | 最小项目构建 | 最小受影响项目能否在目标配置下编译，是否新增错误或警告 | 精确 `dotnet build <project>` 或仓库实际构建命令 |
| L3 | 测试 | 直接相关的单元、组件或回归测试是否覆盖变更及其边界/失败路径 | 精确测试命令、失败/跳过计数、测试报告 |
| L4 | 消费者/集成 | 可定位消费者、模块装配和跨项目契约是否仍能构建并正确集成 | 消费者构建、模块发现/加载、契约兼容结果 |
| L5 | 场景 | 正常、失败、取消、超时、停止、恢复、回滚及人工 UI/授权设备场景是否符合预期 | 自动场景结果或完整人工场景记录 |
| L6 | 全量 | 变更是否在批准的更广仓库范围、配置和回归集合中保持一致 | 解决方案构建、全量测试或批准的回归套件 |

代码变更必须先定位并构建最小受影响项目，不得默认从全解决方案构建开始，也不得用全量输出掩盖最小项目的真实错误。最小项目恢复后，再按调用链扩展到直接测试、消费者和场景；只有影响确实跨越整个解决方案或发布基线要求时才执行 L6。

L2 构建成功只证明编译和相应构建步骤成功，不证明功能、Binding、模块发现、序列化语义、数据库迁移、硬件响应或真实设备行为通过。任何“功能通过”结论必须有对应的 L3-L5 证据。

## 专项验证矩阵

矩阵给出每类变更必须覆盖的内容；实际验证层级仍按最高风险和完整调用链增加。无法执行的项目必须标为“未验证”，不得从矩阵中删除。

| 变更类型 | 最低验证重点 |
| --- | --- |
| C# 业务逻辑 | L0-L3；构建修改项目，运行直接相关测试，覆盖空值、范围、边界、异常输入及失败/取消/超时/释放路径 |
| 公共接口 | L0-L5；定位并构建所有可找到的消费者，验证签名、默认行为、异常/取消语义、二进制或源码兼容，并给出兼容结论 |
| Bug 修复 | 先保留原始症状的可复现证据或说明无法复现原因；再验证修复后症状和回归测试，确认测试在修复前能够暴露问题时才称为回归覆盖 |
| XAML/UI | L0-L3 加 L5；构建页面所属项目，检查页面打开、Binding/命令错误、关键交互、键盘/焦点、长文本、缩放/DPI、主题和关闭/取消路径 |
| Recipe/Output | 验证参数收集、保存、加载、下发、执行后输出刷新、全局/下游缓存刷新、旧方案读取及缺失/无效成员行为 |
| 序列化 | 验证新数据往返、旧数据读取、缺失/额外字段、默认值、枚举数值、错误数据、类型/成员标识和迁移失败恢复 |
| DynamicView | 验证创建、重复打开、多节点隔离、导航往返、删除、反序列化恢复、事件/委托不累积和最终释放 |
| 数据库 | 使用非生产且脱敏的数据验证升级、旧数据读取、事务失败、并发、备份/恢复前提和回滚；生产数据操作必须取得生产数据变更授权，R4 时升级为 R4 执行批准 |
| 硬件 | 优先使用桩/模拟器覆盖离线、超时、异常返回、取消、停止、重连、重复初始化和释放；真实设备验证另需真实设备操作授权与安全前置条件，R4 时升级为 R4 执行批准 |
| 运动控制 | 验证轴映射、单位、方向、速度/加速度、软硬限位、互锁、停止和急停语义；任何真实运动必须取得真实设备操作授权并由人类执行安全方案，R4 时升级为 R4 执行批准 |
| Native 算法 | 验证输入输出契约、边界/非法输入、精度与误差阈值、确定性、性能基线、内存所有权、重复调用和异常释放 |
| 依赖升级 | 单独审查来源、版本、许可证、安全公告、目标架构、API/ABI 兼容、最小项目与消费者构建、运行时复制和回滚版本 |
| 发布配置 | 分别验证 Debug/Release、目标框架/架构、干净输出、模块与依赖复制、模块发现/加载、启动冒烟、停止和回滚；发布动作仍需独立授权 |

## 新鲜证据

新鲜证据必须在最终修改之后、当前环境中产生，并能够与精确变更关联。修改前结果、历史日志、他人结论、缓存的测试报告和“应当通过”不构成本次完成证据；需要引用它们作为基线时，必须明确标记为历史资料。

### 命令证据

每条命令证据必须记录：

- 精确命令及工作目录；
- 执行时间和时区；
- 操作系统、运行时/SDK、配置、架构，以及与结论相关的模拟器或依赖版本；
- 退出码；
- 错误数、失败数和跳过数；工具不提供某字段时写“`不适用`”；
- 支持结论的关键输出，以及完整日志或报告的可复查位置；
- 该命令实际证明的范围和未证明的范围。

禁止只摘录成功行而省略错误、警告、失败或跳过。构建工具未给出测试失败数时，不能用编译错误数代替；人工步骤也不能补造退出码。

### 人工场景证据

每项人工场景证据必须记录：

- 可重复的操作步骤和前置条件；
- 环境、版本/配置，以及模拟器或真实设备身份类别；设备标识必须脱敏；
- 预期结果与实际结果；
- 执行人和执行时间；
- 适用时的截图、日志或测量记录及其脱敏位置；
- 未覆盖的分支和限制。

人工场景没有退出码、错误/失败/跳过计数时，对应字段写“`不适用`”。每个受控操作场景还必须记录授权类型（真实设备、生产数据或其他明确高风险对象）、具体设备或数据目标、操作、授权人及范围、执行人、窗口、检查和停止/回滚/恢复方案；若风险分类为 R4，再关联 R4 执行批准。不得由 AI 代替执行人或批准人，AI 不得执行任何 R4 操作。

## 失败、跳过与未验证

- 本次失败是最终修改后由本次变更引入或仍未满足验收标准的失败，必须调查并阻止相应完成声明。
- 既有失败必须有可复查的基线或独立复现证据，并与本次命令输出同时记录；“看起来是旧问题”不是分类证据。禁止删除、隐藏或重写失败测试来制造成功。
- 跳过项必须说明测试名称、跳过原因、影响和后续负责人；大量跳过不能用零失败包装为完整通过。
- 因缺少依赖、许可证、硬件、数据、权限或执行环境无法运行的验证必须标记“未验证”，说明原因、影响、替代验证和后续负责人。
- 验证命令失败后必须保留原始结果并调查。修正环境或代码后重新运行时，交付中同时说明失败轮次和最终轮次，不得只报告后者来掩盖过程风险。

## 模拟、真实设备与证据边界

- 桩、模拟器、回放、离线数据和静态分析只能证明其模型覆盖的逻辑；必须明确写“模拟验证”或“离线验证”。
- 编译、测试替身和模拟器通过不得表述为真实设备验证，也不能证明设备时序、厂商 SDK、物理限位、急停或工件安全。
- “真实设备验证”仅用于已取得真实设备操作授权、确认设备身份与人员/工件安全，并由具备权限的人类按批准方案执行的记录；风险分类为 R4 时还必须取得 R4 执行批准。
- 没有真实设备或没有授权时，结论必须是“真实设备未验证”，同时列出已完成的模拟层级和残余风险。
- AI 不得连接或操作真实设备，也不得执行 R4 操作；AI 可以整理模拟证据和供人类审查的真实环境验证方案。
