# 架构与依赖边界

## 必须做

- [ ] 明确变更所属层、直接依赖、运行时消费者、数据流和资源所有者。
- [ ] 检查从 `Shell` 装配入口到模块加载、公共契约、配置/数据和输出产物的完整影响链。
- [ ] 确认没有新增反向依赖、循环依赖、UI 对硬件资源的直接所有权或公共层对客户特例的耦合。
- [ ] 列出公共 API、序列化、Recipe、Output、缓存键、数据库和模块标识的兼容影响。
- [ ] 命中本规范的设计评审触发器时，先完成设计、兼容/迁移、验证和回滚评审，再实施行为变更。
- [ ] 按 [测试与验证](testing-and-verification.md) 构建受影响项目与消费者，并验证模块发现、加载、停止和释放路径。

## 分层职责

| 层或目录 | 职责 | 不应承担的职责 |
| --- | --- | --- |
| `Shell/` | 进程入口、启动顺序、Prism/DI 应用装配、全局异常边界、模块目录与主窗口承载 | 供其他层复用的业务逻辑、设备实现、客户特例或可变公共契约 |
| `Core/` | 基础设施、共享抽象、跨模块契约、配置/缓存/数据库基座、公共服务与事件定义；`Core/ReeYin_V.UI` 是当前共享控件、主题、样式和资源字典的权威所有者 | 具体业务流程、厂商设备实现、客户模块或对 `Shell` 的反向依赖 |
| `Application/` | 登录、权限、配置、主业务流程和应用级用例；通过公共契约协调其他能力 | 直接持有厂商 SDK 资源、成为低层基础设施的依赖项 |
| `Hardware/` | 设备抽象、连接与状态契约、厂商适配、超时/停止/重连/释放实现 | 页面业务、客户流程、绕过安全互锁的便捷接口 |
| `Tools/` | 节点工具、执行模块及其参数页面；组合 Core、Hardware、Algorithm 的公开能力 | 复制硬件驱动、直接依赖 `Shell` 或把节点状态散落到 UI 生命周期 |
| `Algorithm/` | 托管/Native 算法、输入输出契约、精度/性能实现和非托管资源封装 | WPF 页面、业务导航、硬件控制或应用装配 |
| `CustomizedDemand/` | 现场或客户专属流程、模块和适配；在公开扩展点上组合标准能力 | 未经设计评审把客户特例、名称或分支下沉到公共层 |
| `CustomUI/` | 项目、业务或客户组合 UI，以及基于 `Core/ReeYin_V.UI` 共享契约的局部扩展 | 全局共享控件/主题/样式的第二权威来源、业务运行流程、设备连接/运动、数据库事务或长生命周期运行资源 |
| `OutputExe/` | 构建输出、运行时模块目录和发布产物的承载位置 | 手工维护的源代码或被源项目引用的稳定输入 |
| `thirdparty/`、`packages/` | 经审查的第三方源码、SDK、二进制和包缓存 | ReeYin-V 业务逻辑、未经来源/许可证/安全审查的依赖投放 |

目录名只表示主要责任，不能替代项目引用和运行时调用链检查。发现现有实现跨越边界时，必须记录兼容约束；不得借当前任务进行无关搬迁，也不得继续扩大违规依赖。

当前共享 UI 的实际所有权以 `Core/ReeYin_V.UI` 为准；`CustomUI` 目录名不表示共享控件、主题或样式已经迁移。`CustomUI` 必须复用现有共享资源。未来迁移共享 UI 所有权必须作为独立的至少 R2 公共契约变更，完成资源/程序集身份设计、全部可定位消费者迁移、兼容窗口、构建与运行时加载验证和回滚；迁移完成并取得新鲜证据前，`Core/ReeYin_V.UI` 的路径、资源键和所有权保持有效。

## 逻辑目标边界与现状基线

本规范的分层职责和依赖方向是**逻辑目标边界**，用于约束新增和修改，不表示当前仓库的全部项目引用已经符合该边界。现有反向引用属于需要兼容处理的 legacy 基线，不能作为新增依赖或职责错位的先例。

截至 2026-07-10，针对审查指出路径的 `csproj` 抽查已确认以下样例：

| 引用方项目 | 被引用项目 | 与目标边界的偏差 |
| --- | --- | --- |
| `Core/ReeYin__V.NodifyUI/ReeYin_V.NodifyManager.csproj` | `Application/ReeYin_V.Share/ReeYin_V.Share.csproj` | Core 目录项目引用 Application |
| `Core/ReeYin-V.Core/ReeYin_V.Core.csproj` | `Tools/Image/ImageTool.Halcon/ImageTool.Halcon.csproj` | Core 引用 Tools |
| `Tools/GeneralALGO/ALGO.BlobAnalysis/ALGO.BlobAnalysis.csproj` | `Shell/ReeYin_V.Shell.csproj` | Tools 引用 Shell |
| `Hardware/Camera/ReeYin_V.Hardware.Camera/ReeYin_V.Hardware.Camera.csproj` | `Application/ReeYin_V.Share/ReeYin_V.Share.csproj` | Hardware 引用 Application.Share |

该表是已确认样例而非完整依赖清单，也不是永久豁免。其他 Hardware、Core、Tools 或业务项目可能存在同类或不同的反向边；最终判断必须以变更时的全部相关 `csproj`、传递引用和代码 usage 检查为准，不能仅对照此表。

对目标边界与 legacy 基线执行以下增量门禁：

- 已有 legacy edge 可以为兼容保留，但只构成现状基线，不授权任何新增引用、usage 或错层职责；未触及的 legacy edge 不要求顺手清理。
- 默认禁止新增反向 `ProjectReference`，也禁止通过 DLL 文件引用、反射或复制类型绕过该门禁。需要新能力时，应把稳定契约放到正确的低层/共享层，由上层提供实现或通过现有扩展点装配。
- 默认禁止沿 legacy edge 新增上层类型、成员或命名空间 usage，默认禁止把新的领域职责下沉到低层或上移到装配/UI 层。

以下任一情况都视为“扩大 legacy edge”：

1. 新增一条从低层/共享层指向上层、具体业务、`Shell` 或客户模块的 `ProjectReference`；
2. 沿现有反向 edge 引用更多上层类型、成员或命名空间，包括新增 `using`、公开签名、属性/特性、DI 注册、反射或序列化类型依赖；
3. 把新的业务、客户、UI 装配或设备实现职责放入逻辑目标边界不拥有该职责的层，即使没有新增项目引用。

“未新增 `ProjectReference`”不能单独证明没有扩大；必须结合代码 usage 和职责归属检查。修改含 legacy edge 的项目时，必须在调查或交付记录中列出项目级 edge、该 edge 的实际代码 usage、此次是否扩大以及迁移方向；该记录只提供审计证据，不构成扩大或例外授权。

仅当用户或任务明确要求扩大，并且在实施前同时满足以下条件时，才能把精确限定的 edge、usage 或职责变化作为受控例外：

1. 形成可评审设计，说明为何不能遵循逻辑目标边界、受影响调用链、替代方案、失败/恢复和回滚；
2. 完成公共 API、序列化、配置、Recipe/Output、缓存、数据库、模块加载和消费者的兼容影响分析；
3. 取得熟悉相关架构和领域的人类负责人评审；作者和 AI 不得自行批准；
4. 按 [例外管理](review-and-delivery.md#例外管理) 记录明确批准人、批准范围、有效期限、残余风险和迁移方向。

受控例外只授权记录中明确的 edge、类型/成员/命名空间 usage 或职责范围，不能沿用为其他新增依赖的先例。安全、人身、设备、权限、凭据和生产数据门禁不得作为 legacy 依赖例外被削弱或绕过；R3/R4 的安全评审、回滚/停止方案、负责人批准和风险披露仍不可豁免。

迁移方向必须指向逻辑目标边界，例如把稳定接口/DTO 下沉到适当共享契约层、由上层注入实现，或把错位业务逻辑移回拥有该职责的模块。迁移必须保持兼容并单独评审，例外到期时必须按批准记录完成迁移、重新批准或停止依赖该例外的交付/运行。

## 依赖方向

### 允许方向

- `Shell` 可以引用装配所需的 `Core` 与启动模块入口；模块通过 Prism/DI 注册、模块目录和稳定契约接入，不把 `Shell` 当共享库。
- `Application`、`Tools`、`Hardware`、`Algorithm` 和 `CustomUI` 可以引用 `Core` 中与自身职责相符的稳定抽象；低层不得因此引入具体业务类型。
- 硬件厂商实现可以依赖同类硬件基座/契约和经审查的厂商 SDK；业务与节点模块只消费硬件抽象，不消费厂商对象。
- `Tools` 可以依赖 `Hardware`、`Algorithm` 暴露的公开契约以组合节点能力；依赖必须指向抽象或受控适配入口，并保留失败、取消、停止与释放语义。
- `CustomizedDemand` 可以在评审后的公开扩展点上依赖 `Core`、`Application`、`Tools`、`Hardware`、`Algorithm` 或 `CustomUI`；被依赖方向不得反转。

### 禁止方向

- 禁止 `Core` 依赖 `Shell`、具体 `Application`、`Tools`、`Hardware` 厂商实现、`CustomizedDemand` 或客户类型。
- 禁止 UI（包括 `Shell` 页面、业务页面和 `CustomUI` 控件）直接创建、连接、使能、运动或拥有相机、控制卡、PLC、IO 等硬件资源；UI 必须调用具有明确所有者和安全边界的服务。
- 禁止公共层、标准模块或共享 UI 反向依赖 `CustomizedDemand`；客户特例进入公共层前必须抽象为通用需求并完成设计、兼容和领域评审。
- 禁止业务模块绕过公共契约直接使用其他模块的内部类型、私有缓存、数据库表实现或输出目录中的 DLL。
- 禁止源项目依赖 `OutputExe` 中的构建产物；禁止把 `thirdparty`/`packages` 当作可随意修改的业务源码目录。
- 禁止为消除编译错误复制公共类型、形成双向项目引用或使用反射访问另一模块内部实现；必须修正契约归属或提交设计评审。

新增项目引用前必须列出“调用方需要什么能力、契约应由谁拥有、运行时由谁实现、失败时如何停止和回滚”。若无法给出单向依赖，必须先设计，不得先加引用再解释。已有反向引用按“逻辑目标边界与现状基线”的增量门禁处理，不能把目标边界误读为要求在无迁移设计时立即删除兼容引用。

## 资源所有权

资源遵循“当前对象创建并保留所有权，或通过明确契约接管所有权，才由当前对象负责释放”的原则。所有权转移必须显式，不能仅凭获得了对象引用推断所有权：

- 硬件连接、采集流、运动控制会话和厂商句柄由 `Hardware` 服务或明确的运行编排器拥有；UI 只订阅状态和发出经校验的命令。
- 数据库连接/事务、文件流、计时器、后台任务、取消令牌源、事件订阅、图像缓冲和 Native 指针必须有唯一可定位的所有者、终止条件和幂等释放路径。
- DI/Prism 注册必须选择与资源寿命一致的作用域。单例不得捕获短生命周期对象；瞬时对象不得启动无人负责的长期任务或永久事件订阅。从 DI 容器解析对象不表示消费者接管所有权，默认由容器/作用域按注册契约管理释放。
- 通过属性 getter、方法参数、服务解析或普通返回值取得的对象默认是借用，消费者不得释放；工厂或返回值只有在契约明确写明所有权转移时才由消费者释放。
- 跨模块传递可释放对象时，契约必须说明调用方是借用、共享还是接管所有权；未说明时不得由消费者擅自释放或长期缓存。共享或引用计数资源必须按契约获取/释放引用，不能把释放底层资源等同于释放当前引用。
- 停止、取消、超时、模块卸载、方案切换和应用退出必须收敛到同一资源终止语义；异常路径不得跳过释放或把硬件留在不确定状态。
- 事件发布者与订阅者不应互相隐式延长寿命。需要取消订阅时保存原委托或订阅令牌，禁止用新建匿名委托伪装取消。

节点 Model/ViewModel 的具体初始化与释放顺序见 [模块开发与生命周期](module-development.md)，通用实现规则见 [编码标准](coding-standards.md)。

## 跨模块契约

- 跨模块调用优先使用由稳定低层拥有的接口、DTO、事件和服务注册；契约不得暴露厂商 SDK 类型、页面控件实例或另一模块的内部缓存对象。
- 契约必须定义输入校验、返回状态、异常边界、取消/超时、线程上下文、重试幂等性和资源所有权；硬件失败不得被折叠为普通成功。
- Prism 导航名、模块名、服务键、事件负载和动态加载标识都是运行时契约。重命名或改变注册时机前必须定位所有消费者及旧配置。
- 跨模块事件只传递完成业务语义所需的数据。发布方不得假定订阅顺序；订阅方不得依赖未声明线程或把异常传播到不拥有的发布循环。
- 公共契约的实现可以替换，但替换后必须保持已承诺行为、失败语义和兼容窗口；不能只以“编译通过”证明运行时兼容。

## 公共 API 与数据兼容

下列内容均视为兼容边界；未经任务明确批准和设计评审，默认保持兼容：

### 公共 API

- 保持公开类型、成员名称、签名、可见性、默认行为和异常语义。新增优先于破坏性修改，但新增也必须检查二义性、默认值和旧实现。
- 改变接口、基类虚成员、泛型约束、DTO 或事件负载时，必须定位并构建所有可找到的消费者；需要破坏性变更时提供迁移顺序、兼容窗口和回滚方案。

### 序列化、Recipe 与 Output

- 保持序列化类型标识、成员名称、类型、默认值、枚举数值和忽略/包含属性。新旧数据必须验证往返、旧数据读取与缺失字段默认值。
- 保持 `[RecipeParam]` 的成员路径、名称和应用时机；不得与导致收集器跳过成员的 `[JsonIgnore]` 组合。
- 保持 `[OutputParam]` 的名称、类型、可写性和下游刷新语义；改变输出必须验证选择、执行更新、全局/下游缓存和旧方案读取。
- 迁移不得仅在内存中“修正”数据而没有持久化、失败恢复和版本识别策略。

### 缓存键与数据库

- 缓存键的命名空间、大小写、格式和生命周期属于公共契约。节点缓存继续使用约定的 `{serial:D3}` 格式、`TryGetValue` 和类型检查；禁止并存两个无迁移关系的键格式。
- 数据库表、列、约束、默认值、加密/连接配置和事务边界不得静默改变。模式变化必须有可重复迁移、旧数据验证、事务失败路径、备份/恢复前提和回滚说明。
- 禁止使用生产数据试迁移，禁止在未授权情况下修改或清理数据库。测试夹具必须脱敏且能证明对应版本边界。

## 模块加载与输出产物

- 模块入口负责注册自身视图、服务和导航类型；不得依赖另一个模块尚未声明的初始化副作用。重复发现、重复打开或反序列化不得累积重复注册和委托。
- 保持模块程序集名、Prism 模块/导航标识、分类、依赖、互斥、站点和许可证规则的兼容；改变任何一项都必须验证禁用、缺失依赖、加载失败和恢复路径。
- 运行时模块产物位于 `OutputExe/<Config>/net8.0-windows/Modules`。构建复制逻辑属于发布契约；必须分别检查 Debug/Release 目标、必要依赖和陈旧产物风险。
- `OutputExe` 是生成/发布区域，不得手工编辑来掩盖源项目、引用或复制规则问题。只有任务明确要求且验证方案覆盖生成来源时才更新产物。
- 模块加载验证至少覆盖：干净输出下生成、模块可发现、依赖满足时加载、禁用或缺依赖时安全失败、应用停止/模块释放时资源关闭。编译成功不能代替运行时发现验证。
- `thirdparty`/`packages` 的新增或替换必须审查来源、版本、许可证、安全、目标架构与部署复制；不得提交真实许可证、密钥或仅在个人机器存在的绝对路径依赖。

## 设计评审触发器

修改前先使用 [变更影响地图](change-impact-map.md) 定位职责、契约和典型消费者；地图是搜索起点，不能替代实际项目引用、代码 usage、动态加载和资源路径调查。

出现以下任一情况，实施前必须形成可评审设计：

- 新模块、新项目、跨层职责迁移、依赖方向或资源所有权变化；
- 公共 API、数据模型、序列化、Recipe、Output、缓存键、数据库或动态加载契约变化；
- MVVM/节点生命周期、线程、并发、取消、重试、停止或释放语义变化；
- 硬件、运动、报警、互锁、权限、安全、网络、外部进程或生产数据变化；
- 破坏兼容、需要迁移/特性开关/分阶段发布，或无法提供安全回滚；
- 存在多个可行方案，且取舍会影响行为、消费者、验证或运维。

设计必须写明目标与非目标、职责和依赖、调用/数据流、契约与所有权、兼容/迁移、失败/恢复、安全控制、验证矩阵、运行限制和回滚。风险和评审要求按 [贡献流程](../../CONTRIBUTING.md) 取最高等级执行。
