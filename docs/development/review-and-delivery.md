# 评审与交付

## 必须做

- [ ] 交付前逐行自审最终差异，确认只包含任务范围，且问题、成功标准、兼容边界、安全路径和回滚要求均有对应证据。
- [ ] 按最高风险等级确定评审和批准所有者；R2 公共契约/兼容边界取得人类领域评审，R3 还须领域评审通过和变更负责人批准，R4 再取得覆盖具体设备或数据目标、操作、执行人、窗口、检查及停止/回滚/恢复的执行批准。
- [ ] 提供最终修改后的新鲜验证证据，分开记录命令证据和人工场景证据，并如实披露失败、跳过和未验证项。
- [ ] 明确公共 API、配置、数据、Recipe、Output、缓存、UI、模块加载和设备行为的兼容结论。
- [ ] 提供可执行的回滚步骤、触发条件、回滚前提、备份/恢复要求、验证方式和回滚负责人。
- [ ] 例外批准只允许限期延期非安全验证，必须记录原因、范围、替代验证、风险、批准人、有效期限和补验期限，且不能替代领域评审、变更批准、受控操作授权或 R4 执行批准。
- [ ] 提交、推送、合并、发布、部署和受控操作分别取得明确授权；真机操作需真实设备操作授权，生产数据操作需生产数据变更授权，相关风险为 R4 时升级为 R4 执行批准。

本规范细化根级 [交付门禁](../../AGENTS.md) 与 [统一贡献流程](../../CONTRIBUTING.md)。验证证据要求见 [测试与验证](testing-and-verification.md)，安全与不可豁免门禁见 [安全与信息安全](safety-and-security.md)，AI 的评审和授权边界见 [AI 开发规范](ai-development.md)。

## 评审与批准术语

- **领域评审**：由熟悉相关技术、安全或兼容边界的人类作出 sign-off 审查结论，回答方案和证据是否满足领域门禁；它不授权交付、合入或真实操作。
- **变更批准**：由明确的变更负责人在领域评审通过后，允许变更在限定范围内交付、合入或进入下一阶段/受控验证；它不授权任何 R4 受控操作。
- **受控操作授权**：按对象分为真实设备操作授权、生产数据变更授权，以及任务明确要求的其他高风险对象授权；必须限定具体设备或数据目标、操作、执行人、窗口、检查及停止/回滚/恢复。
- **R4 执行批准**：风险分类为 R4 时，由相关受控操作授权升级而来并执行 R4 额外门禁；它不能扩大为其他对象、动作或窗口，AI 不得执行。
- **例外批准**：只允许在有效期内缩短或延期非安全验证，并要求限期补验；它不能替代领域评审、变更批准、受控操作授权或 R4 执行批准。

## 评审所有权

评审责任随变更风险和领域归属增加，作者自审不能替代所需的人类领域评审：

| 风险等级 | 最低评审要求 |
| --- | --- |
| R0 | 作者完成内容、路径、链接和最终差异自审；需要时由文档/领域维护者确认事实准确性 |
| R1 | R0 加修改项目、直接测试和局部行为自审；由对应代码/界面所有者评审实现与用户行为 |
| R2 | R1 加消费者、兼容性和场景证据；涉及公共契约或兼容边界时，必须由熟悉该领域的人类评审 |
| R3 | R2 加失败/恢复路径、停止/回滚方案；对应人类领域评审通过并取得明确的变更负责人批准后，才能交付或进入受控验证 |
| R4 | R3 加真实环境验证方案，并取得覆盖具体设备或数据目标、操作、执行人、窗口、检查及停止/回滚/恢复的 R4 执行批准；AI 不得执行任何 R4 操作 |

风险等级取最高值。跨领域变更必须由所有实质受影响领域共同承担评审，例如公共 API 与硬件同时变化时，既需要契约/消费者评审，也需要硬件安全评审。领域评审、变更批准、受控操作授权/R4 执行批准和例外批准分别记录，任何一种都不能替代另一种。作者不得批准自己的例外；AI 不得作为人类领域评审人、批准人或风险接受人。

评审人必须获得足够上下文：问题与成功标准、风险等级、批准设计、完整差异、调用链与消费者、验证证据、未验证项、兼容结论、停止/回滚方案和残余风险。意见解决后若行为、范围或风险发生变化，必须重新运行受影响验证并请求相关部分复审。

## 差异自审

交付前必须基于最终文件逐项检查，而不是依赖编辑过程中的记忆：

- 文件清单是否只包含批准范围；是否夹带无关重构、格式化、依赖、生成物、移动或旁支修复；
- 问题/根因或实现假设、成功标准和明确非目标是否与差异一致；
- 调用方、被调用方、公共消费者、配置/序列化/Recipe/Output/缓存/数据库和模块加载路径是否覆盖；
- 正常、边界、失败、取消、超时、停止、恢复、重试、并发和资源释放是否按实际影响处理；
- API、数据、UI、模块和设备行为是否保持兼容，破坏性变化是否有明确授权、迁移顺序和兼容窗口；
- 日志、截图、命令输出和测试数据是否脱敏；权限、TLS、报警、急停、限位、互锁和警告策略是否未被削弱；
- 验证是否在最终修改后运行，完整输出是否包含退出码、错误/失败/跳过数，既有失败和未验证项是否如实披露；
- 文档、注释、迁移、运行限制和回滚是否与实际实现一致。

自审发现问题后必须修正并重新运行受影响验证。没有最终修改后的新鲜证据，不得进入确定性完成声明。

## 兼容性结论

交付必须逐项给出“兼容”“有批准的变化”或“未验证”，并附依据；不能用笼统的“无影响”覆盖未调查范围。至少考虑：

- 公共类型、成员、签名、可见性、默认行为、异常/取消/超时语义和可定位消费者；
- 配置键、默认值、发布配置、权限、模块名、导航名、服务键和运行时复制；
- 序列化类型/成员、枚举数值、旧数据读取、缺失字段、Recipe 收集/下发、Output 刷新和缓存键；
- 数据库表/列/约束/事务、迁移版本、备份恢复和旧数据；
- UI 页面打开、Binding、交互、缩放/主题、关闭/取消和用户工作流；
- 硬件抽象、厂商实现、报警、离线/超时/停止/重连和真实设备状态。

R2 及以上必须覆盖消费者和兼容路径。无法定位全部消费者时必须说明搜索范围、未知消费者风险和后续负责人；不得把仓库内消费者构建通过扩大为外部消费者兼容。

## 回滚与恢复

回滚说明必须可由具备权限的人按步骤执行，并包含：

- 触发条件和决策人；
- 要回退的精确文件、版本、配置、依赖或迁移；
- 执行前提、服务/模块/设备停止要求和授权；
- 数据、配置和产物的备份位置类别、完整性检查与恢复顺序；
- 公共契约、数据库和序列化变更的向前/向后兼容窗口；
- 回滚后的构建、测试、启动、数据或设备状态验证；
- 回滚失败时的安全停止与升级路径；
- 数据恢复要求、不可逆部分和残余风险。

“恢复旧文件”不是充分回滚方案。若变更写入数据库、生产数据、设备状态、外部系统或不可逆格式，必须说明如何恢复对应副作用；不能安全回滚时必须在实施前升级设计与风险等级并取得负责人决定。

## 例外管理

例外批准只用于在确实无法执行时限期缩短或延期非安全验证，不能降低风险等级、隐藏失败、扩大授权、把“未验证”改写为“通过”，也不能替代领域评审、变更批准、受控操作授权或 R4 执行批准。每个例外记录必须包含：

- 未执行门禁及原因；
- 精确适用范围和明确不包含的范围；
- 已执行的替代验证及其证据；
- 当前影响、已知风险和残余风险；
- 建议补验环境、补验内容和后续负责人；
- 具备相应权限的人类例外批准人；
- 批准生效时间和有效期限；
- 补验期限，以及逾期撤销例外或停止交付/运行的动作。

以下项目不可豁免：人身、设备、权限、凭据和生产数据安全底线；R3/R4 的领域/安全评审、停止/回滚方案、变更负责人批准和风险披露；R4 的具体执行批准，且 R4 不由 AI 执行。紧急情况下只可以缩短或延期非安全验证，仍必须完整记录上述字段。AI 可以起草和上报例外请求，不得批准例外、接受风险或把例外当作永久先例。

## 独立授权动作

代码或文档实现、评审、构建和测试完成，不自动授权下列动作：

- 创建提交、修改 Git 历史或打标签；
- 推送、开启/更新合并请求或合并；
- 发布包、复制发布产物、部署或修改发布配置；
- 修改、迁移或删除生产数据，并取得生产数据变更授权；
- 连接、使能、驱动或操作真实设备，并取得真实设备操作授权。

每项动作都必须由具备权限的用户或负责人单独明确授权，并遵守对应安全、数据和回滚门禁。AI 不得自行执行或批准这些动作。
