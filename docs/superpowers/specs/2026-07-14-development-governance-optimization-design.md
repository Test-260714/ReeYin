# ReeYin-V 开发治理优化设计

**状态：** 用户已确认，已进入实施  
**日期：** 2026-07-14  
**风险等级：** R2（改变仓库开发流程、风险判定入口和最低文档门禁，但不降低现有安全底线）

## 1. 背景

仓库已经建立根级 `AGENTS.md`、统一贡献流程、专题规范和五个领域级 `AGENTS.md`。现有体系对安全、真实设备、生产数据、AI 权限、验证证据和 ReeYin-V 模块生命周期规定较完整，但简单代码修改仍可能承担与高风险变更相近的阅读、调查和报告成本。

本次优化不通过降低 R2-R4 门禁提速，而是建立可审计的低风险快速通道、从修改位置到消费者和验证范围的影响导航、按需阅读机制及确定性的自动化检查。

## 2. 目标

- 让贡献者从修改文件快速判断必须调查的契约、消费者、副作用和验证范围。
- 为纯文档和单项目私有局部修改提供严格准入的 `R0-Lite`、`R1-Lite` 通道。
- 保留公共契约、生命周期、持久化、硬件、安全、权限、生产数据和发布变更的现有门禁。
- 减少风险定义和通用条款在多个文件中的重复，建立单一权威来源。
- 补足高风险一级目录的就近规则，并为依赖、资源和生成物目录提供保护规则。
- 把可客观判断的治理要求自动化，但不让脚本替代风险、评审或安全判断。

## 3. 非目标

- 不指定固定个人、账号或团队作为预设负责人。
- 不自动批准风险等级、评审、例外、真实设备操作或生产数据操作。
- 不构建完整的静态调用图或声称自动发现全部动态消费者。
- 不改变产品运行代码、公共 API、序列化、Recipe、Output、缓存、数据库或设备行为。
- 不在本次治理优化中清理历史代码、依赖、生成物或无关文档。

## 4. 设计原则

1. **风险决定成本**：低风险变更只承担证明其低风险所需的最小成本；高风险变更维持完整门禁。
2. **影响优先于目录名**：目录提供调查起点，最终等级由实际契约、消费者和副作用决定。
3. **单一权威来源**：风险等级与统一流程由 `CONTRIBUTING.md` 权威定义，其他文档只摘要并链接。
4. **自动化不替代判断**：脚本检查文件、链接、结构和必填字段，不判断业务风险是否可接受。
5. **局部规则只写增量**：领域 `AGENTS.md` 只写该目录特有职责、调查点、升级条件和验证要求。
6. **发现扩大立即升级**：快速通道中出现未知消费者、公共契约或高风险副作用时，停止 Lite 流程并重新分级。

## 5. 目标规则结构

### 5.1 根级入口

根 `AGENTS.md` 保留：

- 规则优先级和就近规则继承关系；
- 安全、权限、真实设备、生产数据和 AI 不可越过的底线；
- 保护用户改动、最小范围和新鲜验证证据要求；
- 指向 `CONTRIBUTING.md`、影响地图和专题规范的入口。

根文件不再重复完整风险矩阵和专题实现细节。

### 5.2 统一流程

`CONTRIBUTING.md` 是以下内容的唯一权威来源：

- `R0-Lite`、`R1-Lite`、R1-R4 的定义；
- 风险升级和设计触发器；
- 调查、实现、验证、自审、评审和交付流程；
- 规则自身变更的治理要求。

### 5.3 导航和专题规范

- `docs/development/README.md`：按变更类型和风险指向必须阅读的文件或章节。
- `docs/development/change-impact-map.md`：从目录或变更类型定位职责、契约、典型消费者和升级条件。
- `docs/development/build-and-test-map.md`：提供最小构建、直接测试、主要消费者和特殊环境入口。
- 现有专题文档只保留各自独有的架构、编码、验证、安全、评审、AI 和生命周期细节。

### 5.4 就近目录规则

新增完整领域规则：

- `Application/AGENTS.md`
- `Shell/AGENTS.md`
- `CustomizedDemand/AGENTS.md`
- `Semiconductor/AGENTS.md`

新增简短目录保护规则：

- `thirdparty/AGENTS.md`
- `packages/AGENTS.md`
- `OutputExe/AGENTS.md`
- `Resource/AGENTS.md`

实施前检查 `GemeralUI/` 的项目职责、资源所有权和消费者，再决定创建独立规则还是在现有 UI 领域规则和影响地图中明确其归属。不得根据目录名直接推断。

每个新增领域规则只包含：适用范围、职责与公共契约、修改前调查、风险升级条件、最低验证和交付要求。

## 6. 快速通道

### 6.1 R0-Lite

适用于错别字、注释和纯说明文档，且不得改变开发步骤、构建方式、配置、公共契约、运行行为、安全要求或治理门禁。

最低动作：

1. 检查工作区和适用规则；
2. 检查最终差异、内容准确性、相对路径、链接和敏感信息；
3. 简要报告修改范围和未涉及的运行行为。

改变流程、风险定义、安全要求或验证最低要求的文档不属于 R0-Lite。

### 6.2 R1-Lite

只有同时满足以下全部条件时才可使用：

- 修改限制在一个项目；
- 不改变 `public` 或 `protected` API；
- 不改变配置、序列化、Recipe、Output、缓存键或数据库；
- 不改变异步契约、线程切换、取消、重试、资源所有权或生命周期；
- 不涉及硬件命令、报警语义、权限、安全、发布或生产数据；
- 不新增或升级依赖；
- 不改变模块注册、动态加载、导航身份或共享 XAML 资源；
- 调用方、影响边界和直接测试能够明确定位；
- 不与用户现有修改发生无法安全处理的重叠。

最低动作：

1. 检查工作区、根规则、最近的 `AGENTS.md` 和贡献流程快速清单；
2. 定位修改入口、直接调用方、直接测试和失败路径；
3. 执行最小修改；
4. 构建修改项目并运行直接相关测试；
5. 用紧凑证据报告风险理由、命令结果、兼容结论和回滚方式。

命中任一排除项、发现未知消费者或风险扩大时，必须退出 `R1-Lite`，重新按 R1-R4 分类。

### 6.3 R1-R4

- R1：多文件内部行为、异步、线程、资源释放，或缺少可定位直接测试的内部变化。
- R2：公共 API、MVVM 生命周期、Recipe、Output、序列化、缓存、数据库及其他兼容边界。
- R3：硬件运动、报警语义、权限、安全、迁移和发布配置。
- R4：对设备、人身或生产数据存在不可逆风险的受控操作。

R2-R4 延续现有消费者、兼容、领域评审、安全、停止、回滚、授权和真实环境边界，不因引入 Lite 通道而缩减。

## 7. 按需阅读模型

| 风险/通道 | 最低阅读范围 |
| --- | --- |
| 任意变更 | 根规则、最近的 `AGENTS.md`、贡献流程快速清单 |
| R0-Lite | 文档、路径、链接和敏感信息相关条款 |
| R1-Lite | 目标领域规则和影响地图指向的相关专题章节 |
| R1 | 与实际调用链、线程、异常、释放和 UI 行为相关的专题 |
| R2 | 契约、架构、测试、评审及所有受影响领域规范 |
| R3/R4 | 安全、测试、评审、AI 及所有受影响领域规范全文 |

按需阅读只排除无关文档，不减少实际调用链、消费者、数据、线程、资源和安全影响调查。

## 9. 构建与测试地图

`build-and-test-map.md` 记录：

- 推荐 SDK、操作系统、配置和解决方案入口；
- 项目最小构建命令；
- 直接测试项目及其实际运行方式；
- 主要消费者构建；
- 厂商 SDK、许可证、Native DLL、目标架构和 `SolutionDir` 等特殊前提；
- 命令已验证、仅静态核对或未验证的证据状态。

历史计划中的命令只能作为候选。写入地图前必须通过当前 `.sln`、`.csproj` 和测试入口静态核对；只有在当前环境实际运行并保留新鲜证据后，才能标记为已验证。

未列出测试不代表无需测试。贡献者必须记录测试缺口、影响、替代验证和后续动作。

## 10. 自动化治理检查

新增 `scripts/Test-Governance.ps1`，第一阶段仅执行确定性、只读检查：

- 必要规则文件存在；
- Markdown 相对文件链接能够解析；
- 不存在 `TODO`、`TBD` 和合并冲突标记；
- 一级源码/资源/产物目录在影响地图中有明确覆盖或排除理由；
- 领域 `AGENTS.md` 能追溯到根规则；
- 完整风险等级只由权威流程定义，其他文件使用摘要或链接；
- 规则文件不包含仓库外机器绝对路径或明显凭据模式；
- 交付模板包含风险、验证、兼容、回滚、未验证和评审/批准字段。

脚本返回非零退出码表示客观治理检查失败，并输出精确文件和原因。脚本不得自动修改文件，也不得判断风险是否可接受、消费者是否完整、评审是否有效或设备是否安全。

## 11. 规则自身治理

- 改变安全底线、风险定义、授权边界或验证最低要求，至少按 R2 治理变更处理。
- 删除或放宽领域规则必须说明影响、兼容、替代门禁、验证和回滚。
- 新增一级源码、资源或产物目录时，必须更新影响地图并判断是否需要就近 `AGENTS.md`。
- 规则变更必须运行治理检查，并检查所有摘要、链接和引用是否仍一致。
- 只有不改变语义的措辞、错别字和纯说明修正才可使用 R0-Lite。
- AI 不得批准规则例外、降低风险等级或将治理脚本成功解释为领域评审通过。

## 12. 迁移顺序

1. 建立权威风险定义和 Lite 准入/退出条件。
2. 建立影响地图和构建测试地图，先提供调查入口。
3. 增加缺失的领域和目录保护规则。
4. 调整索引和按需阅读规则。
5. 将专题与领域文档中的重复风险表改为摘要和权威链接。
6. 增加只读治理脚本。
7. 运行链接、结构、重复定义、占位符和最终差异检查。

去重必须在权威来源和映射建立之后进行，避免先删除内容造成门禁丢失。

## 13. 验证方案

本次属于治理文档 R2 变更，实施后至少验证：

- 最终差异只包含治理优化范围；
- 所有新增和修改 Markdown 相对链接可解析；
- 风险等级、Lite 条件、升级条件和不可豁免门禁之间无冲突；
- 每个一级目录在影响地图或明确排除说明中出现；
- 新增 `AGENTS.md` 只收紧或细化根规则；
- 治理脚本在正常仓库状态返回成功，并能通过受控夹具证明断链、缺文件和冲突标记会失败；
- `build-and-test-map.md` 中每条命令标明已验证、静态核对或未验证；
- 对 R0-Lite、R1-Lite、R2、R3 各选一个仓库示例进行纸面演练，确认能得到明确阅读、调查和验证清单；
- 不把文档或脚本验证表述为产品代码、CI 或真实设备验证。

## 14. 合理性与风险分析

### 14.1 合理性

- 提速来自排除无关阅读和无关验证，不来自降低安全要求。
- 影响地图回答“改这里还要调查哪里”，比预先绑定固定负责人更适合当前目标。
- 单一权威来源降低多份风险表长期漂移的概率。
- 局部规则保存领域知识，专题文档保存通用知识，职责更清楚。
- 自动化处理客观、重复的检查，把领域判断留给人类和任务上下文。
- 构建测试地图复用仓库知识，但用证据状态避免把历史命令误报为当前验证。

### 14.2 已知风险与控制

| 风险 | 影响 | 控制措施 |
| --- | --- | --- |
| Lite 通道被误用 | 公共或高风险影响验证不足 | 全部条件同时满足；任一排除项或未知消费者即升级 |
| 影响地图过时 | 漏查消费者或副作用 | 新增一级目录门禁；地图只作为起点；实际代码搜索仍强制 |
| 文档去重误删门禁 | 规则被意外放宽 | 先建立权威来源和旧条款映射，再逐项去重 |
| 新增规则增加阅读负担 | 简单任务仍然缓慢 | 领域规则限制为四类增量信息；支持按章节阅读 |
| 构建命令依赖机器环境 | 地图给出不可执行命令 | 为命令标记证据状态和特殊前提 |
| 自动化产生虚假安全感 | 脚本成功被误认为评审或验证通过 | 明确脚本只证明结构和链接；禁止自动批准风险 |
| 目录职责判断错误 | 规则与真实架构不符 | 创建规则前调查项目引用、代码 usage、注册、数据和资源路径 |

## 15. 回滚

本次只修改治理文档和只读检查脚本。回滚时按文件恢复本次差异，并恢复原风险矩阵和阅读入口；不涉及产品数据迁移、运行配置或设备状态恢复。若部分迁移已经落地，必须先保证 `CONTRIBUTING.md` 中仍存在完整权威风险定义，再撤销摘要或链接，避免出现无权威来源的中间状态。
