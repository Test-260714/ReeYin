# ReeYin-V 开发治理规范设计

## 1. 目标

为 ReeYin-V 建立一套同时适用于人类开发者、AI 开发代理和自动化工具的统一开发规范。规范采用强制门禁，覆盖需求澄清、代码调查、实现、验证、评审和交付全过程，并针对工业软件、硬件控制、模块化架构和 MVVM 生命周期增加领域约束。

规范应达到以下目标：

- 让贡献者在修改前知道必须读取什么、确认什么以及禁止做什么。
- 让完成声明由新鲜、可复核的验证证据支持。
- 让公共 API、序列化、配方、缓存、硬件和安全相关变更接受与风险匹配的审查。
- 让全局规则与目录领域规则分层维护，避免单个巨型规范文件。
- 允许有记录、有批准、有期限的例外，但禁止无说明跳过门禁。

## 2. 适用对象与规范强度

规范适用于：

- 人类开发者和评审者；
- AI 开发代理；
- 执行构建、测试、代码生成或发布准备的自动化工具。

规范用词定义如下：

- **必须**：强制要求，只有经过规定的例外审批才能偏离。
- **禁止**：不得执行，除非更高优先级的明确授权允许且不违反安全底线。
- **应当**：默认要求；偏离时必须说明原因、影响和替代措施。
- **可以**：根据任务范围和风险选择执行。

AI 不具备批准例外、接受生产风险或授权危险操作的权限。

## 3. 设计原则

### 3.1 单一入口，分层细化

根级 `AGENTS.md` 是工具执行入口，`CONTRIBUTING.md` 是统一贡献流程入口，`docs/development/README.md` 是详细规范索引。子目录 `AGENTS.md` 只增加领域规则，不重复根规则，也不得降低根级门禁。

### 3.2 证据先于结论

不得仅依据代码外观、旧的构建记录、局部检查或个人信心宣称任务完成。完成声明必须引用当前变更后的构建、测试或场景验证结果，并披露未验证项。

### 3.3 风险决定验证深度

文档、局部 UI、公共 API、序列化、硬件运动和生产数据变更采用不同最低门禁。一个变更涉及多个类别时采用最高风险等级。

### 3.4 当前模式优先，架构边界不退让

修改现有模块时优先遵循当前项目和相邻文件的实现模式。既有模式若违反安全底线、公共契约或明确生命周期要求，应记录冲突并采用经评审的修复方案，而不是机械复制。

### 3.5 小而独立的变更

功能修改、架构重构、依赖升级和批量格式化应拆分。每个变更应能够独立理解、验证和回滚。

```text
│       └── ai-development.md

├── Hardware/AGENTS.md
├── Algorithm/AGENTS.md
├── Tools/AGENTS.md
└── CustomUI/AGENTS.md
```

### 4.1 根级 `AGENTS.md`

包含适用范围、指令优先级、规范词义、修改前门禁、范围控制、安全底线、验证门禁、交付门禁和 AI 专属权限边界。文件保持简洁，通过链接指向详细规范。

### 4.2 `CONTRIBUTING.md`

定义从问题描述到交付的统一流程，并规定 Bug 报告的最小复现、预期行为、实际行为、环境信息、回归状态、脱敏日志和临时解决方法。

### 4.3 `docs/development/`

- `README.md`：规范地图、快速检查表和风险分级入口。
- `architecture.md`：模块职责、依赖方向、资源所有权和兼容性边界。
- `coding-standards.md`：C#、XAML、异步、异常、日志、事件和资源释放规范。
- `testing-and-verification.md`：验证层级、风险矩阵、命令证据和场景测试。
- `module-development.md`：ReeYin-V 节点、MVVM、配方、输出、缓存和动态页面生命周期。
- `safety-and-security.md`：硬件、运动、生产数据、凭据、网络和外部进程安全。
- `review-and-delivery.md`：评审责任、交付模板、兼容性和回滚要求。
- `ai-development.md`：AI 调查、权限、事实表达、工具使用和完成声明规则。

### 4.4 子目录 `AGENTS.md`

- `Core/AGENTS.md`：公共 API、依赖方向、数据库、缓存、事件、线程和兼容性。
- `Hardware/AGENTS.md`：设备安全、模拟与真机隔离、超时、停止、重连和释放。
- `Algorithm/AGENTS.md`：输入输出契约、精度、性能、确定性和 Native 内存。
- `Tools/AGENTS.md`：节点生命周期、Recipe/Output、执行状态、动态页面和缓存。
- `CustomUI/AGENTS.md`：Prism 导航、Binding、UI 线程、共享样式、缩放和交互。

## 5. 强制工作流

### 5.1 修改前

贡献者必须：

1. 明确问题、预期行为、验收标准和不在范围内的内容。
2. 检查工作区状态，识别并保护已有未提交改动。
3. 定位待修改项目、入口、调用链、数据流和生命周期。
4. 阅读待修改代码及至少一个相关的相邻实现；没有可比实现时记录这一事实。
5. 识别公共 API、序列化、Recipe、Output、缓存、数据库、线程、UI 和硬件影响。
6. 为跨模块、高风险或兼容性变更先形成设计并获得确认。

### 5.2 实现中

贡献者必须控制变更范围，不得夹带无关重构、批量格式化、依赖升级或生成物修改。新增行为应有可验证的失败路径和资源释放路径。旧模块迁移必须优先保持序列化、缓存、输出、窗口和执行流程兼容。

### 5.3 完成前

贡献者必须：

1. 重新逐项核对验收标准。
2. 检查最终差异和新增文件。
3. 运行风险等级要求的构建、测试和场景验证。
4. 阅读完整输出，确认退出码、错误数、失败数和跳过数。
5. 区分本次失败、既有失败和环境限制。
6. 使用统一模板报告修改、验证、未验证项、兼容性、回滚和残余风险。

## 6. ReeYin-V 架构与生命周期门禁

### 6.1 模块边界

- `Shell` 负责启动和应用装配。
- `Core` 提供基础设施与共享契约，不得反向依赖具体业务模块。
- `Application` 承载业务功能。
- `Hardware` 提供设备抽象和厂商实现；业务与 UI 不得绕过抽象直接控制厂商对象。
- `Tools` 承载节点工具和执行模块。
- `Algorithm` 承载算法及 Native 能力。
- `CustomizedDemand` 承载客户定制，不得无评审地把客户特例下沉到公共层。

### 6.2 Model 与 ViewModel

- 持久化节点主 Model 使用 `ModelParamBase`，主 ViewModel 使用 `DialogViewModelBase`。
- 主 ViewModel 通过 `InitModelParam<TModel>()` 初始化完整生命周期。
- `LoadKeyParam()` 默认必须调用 `base.LoadKeyParam()`。
- `OnceInit()` 必须防止重复初始化，并在节点特定注册前调用 `base.OnceInit()`。
- `Dispose()` 必须先释放本地资源和事件，再调用 `base.Dispose()`。
- `TriggerModuleRun` 不得因反序列化或重复打开累积重复委托。

### 6.3 Recipe、Output 与动态页面

- `[RecipeParam]` 不得与导致收集器跳过成员的 `[JsonIgnore]` 组合。
- `[OutputParam]` 必须标记可写成员，并在执行后按需要刷新输出与下游缓存。
- `NodeParamCaches` 使用 `{serial:D3}`、`TryGetValue` 和类型检查。
- NodeMap 页面在模块入口注册导航类型，在 `OnceInit()` 中按节点添加，并依靠正确的释放链移除。
- XAML 优先复用 `ReeYin_V.UI` 共享样式与控件。

详细模式链接到现有 `docs/reeyin-v-mvvm-skill.md`，正式规范不另行发明冲突的生命周期。

## 7. 风险分级与最低验证

| 等级 | 典型变更 | 最低门禁 |
|---|---|---|
| R0 | 文档、注释，不改变运行行为 | 差异、内容、路径和链接检查 |
| R1 | 局部 UI、内部逻辑、非公共重构 | R0 + 修改项目构建 + 相关测试 |
| R2 | 公共 API、MVVM 生命周期、Recipe、Output、序列化、缓存、数据库 | R1 + 消费者构建 + 兼容性和场景测试 |
| R3 | 硬件运动、报警、权限、安全、迁移、发布配置 | R2 + 失败路径 + 回滚说明 + 领域评审 |
| R4 | 设备、人身或生产数据存在不可逆风险 | R3 + 真实环境验证方案 + 明确负责人批准 |

R4 操作不得由 AI 自行执行。

## 8. 专项验证矩阵

| 变更类型 | 强制验证 |
|---|---|
| C# 业务逻辑 | 修改项目构建、相关测试、边界和异常输入 |
| 公共接口 | 所有可定位消费者构建、兼容性说明 |
| Bug 修复 | 原始症状复现、修复后通过、回归测试 |
| XAML/UI | 构建、页面打开、Binding、关键交互、缩放和主题 |
| Recipe/Output | 收集、保存、加载、下发、输出和下游缓存刷新 |
| 序列化 | 新数据往返、旧数据读取、缺失字段默认值 |
| DynamicView | 创建、重复打开、多节点隔离、删除和释放 |
| 数据库 | 升级、回滚、旧数据、事务失败路径 |
| 硬件 | 离线、超时、异常返回、停止、重连和释放 |
| 运动控制 | 单位、轴、速度、限位、互锁、停止和急停 |
| Native 算法 | 内存所有权、边界输入、误差和性能基线 |
| 依赖升级 | 来源、许可证、安全、API 兼容和构建 |
| 发布配置 | Debug/Release、输出、模块复制和启动冒烟 |

## 9. 安全底线

- 未经明确授权不得连接、初始化、运动、复位或改变真实设备状态。
- 运动控制必须核对轴、单位、方向、速度、限位、互锁和停止路径。
- 硬件异常不得被转换为成功结果或仅写日志后继续危险操作。
- 凭据、密钥、客户数据、设备标识和生产配置不得写入源码或未脱敏输出。
- 新增网络请求、遥测、上传、外部进程或脚本执行必须显式评审。
- 不得为通过验证而关闭权限、TLS、安全检查或全局编译警告。
- AI 不得自行提交、推送、发布、部署、批准例外或执行破坏性版本控制命令。

## 10. 例外审批

无法完成门禁时，必须记录：

```text
未执行的门禁：
原因：
影响范围：
替代验证：
残余风险：
建议补验环境：
批准人：
有效期限：
```

R3 和 R4 例外需要明确的人类领域负责人批准。缺少硬件、许可证或依赖只能形成“未验证”结论，不能转化为通过。紧急修复可以缩短流程，但必须保留安全评审、回滚方案和限期补验。

## 12. 规范维护

- 根级文件只保留必须立即执行的规则，详细解释放在专题文档。
- 每份专题文档开头提供可操作的“必须做”检查表。
- 子目录规则只写增量要求。
- 规则应对应命令、测试、场景或明确的人工检查，避免不可验证口号。
- 规则与实际工程冲突时必须记录并修正规范，不得长期口头绕过。
- 公共生命周期或架构变化时同步更新相关规范。
- 新增高风险领域时增加就近规则和负责人。

## 13. 外部项目参考

设计参考以下大型项目的公开贡献规范：

- Kubernetes：单一贡献入口、自动门禁和目录责任边界。
- .NET Runtime：最小复现、预期与实际行为、当前项目风格优先。
- Chromium：分层评审、按风险选择测试、跨组件影响检查。
- Linux Kernel：小而独立的变更、明确子系统、长期可读的“为什么”。

这些项目提供治理原则，不表示 ReeYin-V 采用其组织、签署或提交工具流程。ReeYin-V 的最终规则以工业设备安全、模块化加载、WPF/Prism、MVVM 生命周期、配方、输出和缓存一致性为核心。

参考链接：

- <https://github.com/kubernetes/community/tree/master/contributors/guide>
- <https://github.com/dotnet/runtime/blob/main/CONTRIBUTING.md>
- <https://chromium.googlesource.com/chromium/src/+/main/docs/contributing.md>
- <https://github.com/torvalds/linux/blob/master/Documentation/process/submitting-patches.rst>

## 14. 验收标准

正式规范实施完成后应满足：

1. 根目录存在可被开发代理直接发现的 `AGENTS.md`。
2. 人类贡献者可以从 `CONTRIBUTING.md` 找到完整流程。
3. 所有专题文档和子目录规则链接可解析。
4. 强制门禁、风险等级、例外审批和交付模板不存在相互冲突。
5. ReeYin-V 生命周期规则与现有框架规范一致。
6. 文档明确区分模拟验证、真实设备验证和未验证结论。
7. 文档不包含未解决占位符或依赖口头约定的关键要求。
