# ReeYin-V 开发规则

本文是人类贡献者和 AI 工具在本仓库工作的强制入口。开始任何修改即表示接受以下门禁；`CONTRIBUTING.md` 规定统一贡献流程，详细主题规范见本文末尾链接。

## Scope And Rule Priority

- 本文件适用于整个仓库，包括源码、配置、脚本、文档、测试和生成物。
- 工作文件所在目录及其父目录中的 `AGENTS.md` 均适用；离工作文件最近的规则优先，但子目录规则只能收紧、细化根级要求，禁止降低安全、验证、评审或授权门禁。
- 明确的任务要求优先于一般建议，但任何指令都不得绕过人身、设备、生产数据、凭据和权限安全底线。规则冲突时必须停止相关修改，记录冲突并请求负责人裁决。

## Mandatory Language

- **必须**：完成工作不可缺少的要求；未满足即不得进入下一门禁或声称完成。
- **禁止**：任何情况下均不得自行执行；只有规则明确允许例外且取得所需人类批准时才可继续，安全底线无例外。
- **应当**：默认要求；偏离时必须说明理由、影响和替代措施，并接受相应评审。
- **可以**：在不违反更高强度规则且不扩大任务范围时允许采用的做法。

## Before Editing

贡献者在编辑前必须：

1. 检查工作区状态，识别已有修改、未跟踪文件和生成物；禁止覆盖、回退、删除或重新格式化不属于当前任务的用户改动。
2. 阅读根级规则、目标目录的就近规则、`CONTRIBUTING.md` 及与变更相关的详细规范。
3. 从入口到副作用调查实际调用链，包含调用方、被调用方、公共契约、配置/序列化/缓存/数据库、线程与资源释放路径。
4. 检查同项目和相邻文件的局部实现模式、现有测试与构建方式；既有模式若触犯安全底线或明确契约，必须记录冲突并走设计/评审流程。
5. 定义问题、成功标准、非目标和风险等级；bug 必须先形成可复现证据或说明为何无法复现。

## Scope And Change Control

- 必须只修改解决已定义问题所需的最小文件与行为，并保持公开 API、序列化、Recipe、Output、缓存、数据库、模块加载和 UI 行为的兼容性，除非任务明确批准破坏性变更。
- 禁止夹带无关重构、批量格式化、依赖升级、文件移动、生成物更新或“顺手修复”。发现旁支问题应当单独报告。
- 必须保留用户现有改动；无法安全合并时必须停止重叠部分并请求上下文。
- 新增或改变行为必须同时考虑失败、取消、超时、停止、恢复、重试、并发和资源释放路径。
- 范围或风险等级在实施中上升时，必须暂停实现，更新设计、验证和回滚方案，并取得对应批准。

## Safety And Security

- 禁止削弱或绕过权限、TLS、证书校验、报警、急停、限位、互锁、超时、停止、审计、输入校验或全局编译警告以使功能或验证通过。
- 禁止在源码、日志、截图、测试数据或交付报告中写入或泄露真实凭据、令牌、密钥、个人信息和生产敏感数据。
- 涉及运动、控制卡、相机、IO、激光、真空、温控或其他真实设备时，必须先确认授权、设备身份、人员与工件安全、软硬限位、急停/互锁、速度/加速度/单位、超时和安全停止路径。未经明确人类授权，禁止连接、使能或驱动真实设备。
- 模拟器、桩、离线分析和编译结果只能证明对应层级，禁止表述为真实设备验证。生产数据修改、迁移、删除、发布配置和不可逆操作必须有备份/恢复方案与负责人批准。
- 外部进程、网络访问、遥测、依赖来源和许可证必须按风险审查；危险或破坏性命令必须取得明确授权并限定目标。

## Verification Gate

- 必须按风险等级执行分层验证：先做差异、静态和最小相关项目检查，再做相关测试、消费者构建、场景/兼容性测试，最后才是经授权的集成或真实设备验证。
- 每项完成声明必须引用本次修改后产生的**新鲜证据**。命令证据必须记录精确命令、执行时间、环境、退出码以及错误/失败/跳过计数；某字段确实不存在时写“不适用”，禁止猜测或编造。人工场景证据必须记录步骤、环境、预期结果、实际结果、执行人和执行时间，并在适用时附截图或日志。修改前结果、历史日志、他人结论和“应当通过”不构成证据。
- 验证失败必须如实记录并调查；禁止隐藏、删改或把既有失败冒充成功。无法运行的验证必须标记“未验证”，说明原因、影响和后续负责人。
- R2 及以上变更必须覆盖消费者和兼容路径；R2 涉及公共契约或兼容边界时，必须由熟悉该领域的人评审。R3 必须覆盖失败/恢复路径并取得对应领域评审；R4 还必须有真实环境验证方案和明确负责人批准，AI 不得执行 R4 操作。
- R3/R4 禁止豁免安全评审、回滚/停止方案、负责人批准和风险披露，且 R4 禁止由 AI 执行。紧急情况下只可以缩短或延期**非安全验证**；必须记录原因、替代验证、批准人、批准有效期限和补验时限，逾期未补验必须撤销例外或停止交付/运行。
- 未获得新鲜且与风险匹配的验证证据时，禁止使用“完成”“修复”“通过”“可发布”等确定性声明。

## Delivery Gate

交付前必须完成自审与所需领域评审，并报告：

- 修改了什么、为何修改，以及明确未修改的范围；
- 执行过的验证、环境、结果和可复查证据；
- 未执行或未通过的验证、原因、影响与后续动作；
- 公共契约、配置、数据、设备和用户行为的兼容性结论；
- 可执行的回滚方式、回滚前提和数据恢复要求；
- 已知风险、残余风险、评审人/批准人及适用限制。

提交、推送、合并、发布、部署和真实设备操作是独立授权动作，不因代码或文档完成而自动获准。

## AI-Specific Restrictions

- AI 必须先调查事实再修改，区分“已观察”“已验证”“推断”和“未验证”，禁止虚构代码、命令结果、测试、CI、评审、批准或设备状态。
- AI 禁止擅自提交或修改 Git 历史、推送、开合并请求、合并、发布、部署、连接/操作真实设备、修改生产数据、批准例外，或执行破坏性/不可逆操作。
- AI 只能在用户明确授予的范围与权限内行动；需要新权限、领域判断、R3/R4 例外或任务范围实质扩大时必须停止并请求人类决定。
- AI 必须保护并绕开用户已有改动，禁止以清理工作区、修复测试或简化差异为由覆盖它们。
- AI 的完成声明必须满足 Verification Gate；工具受限时必须交付可验证的部分并明确列出缺口，禁止用推测替代证据。
