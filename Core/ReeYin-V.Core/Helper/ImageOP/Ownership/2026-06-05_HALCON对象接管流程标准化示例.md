# HALCON 对象接管流程标准化示例

> 目的：把 `HalconImageOwnership` 这套“借来的先复制，自己的才释放”的做法，整理成后续改造其他 HALCON 节点时可照抄的流程示例。后面如果要做成 Codex Skill，可以直接从本文抽取触发条件、排查步骤、改造模板和验收清单。

## 1. 本次仓库对照结论

本次优先对照了三个层次：

- 新做法主样板：`Tools/GeneralALGO/ALGO.LineScanSheetCounter/Models/LineScanSheetCounterModel.cs`
- 区域对象主样板：`Tools/GeneralALGO/ALGO.RegionProcessing/Models/RegionProcessingModel.cs`
- 旧做法参考：`Tools/GeneralALGO/ALGO.CreatRegion/Models/CreatRegionModel.cs`、`Tools/GeneralALGO/ALGO.RegionTrans/Models/RegionTransModel.cs`、`Tools/GeneralALGO/ALGO.DistanceLL/Models/DistanceLLModel.cs`

一句话结论：线扫模块适合当“图像接管 + 算法结果 Detach”的样板；区域处理模块适合当“区域 HObject 输出 + 旧输出退休释放”的样板；旧模块适合当反面对照，不建议直接照搬。

## 2. 统一类的定位

所属模块与文件路径：`Core/ReeYin-V.Core/Helper/ImageOP/HalconImageOwnership.cs`

`HalconImageOwnership` 不是“大管家登记所有对象”的注册中心，它更像一个安全钳子：

- 判断对象活性：`IsInitializedSafe(...)`
- 判断图像是否真能读尺寸：`TryGetImageSize(...)`
- 把 borrowed 图像复制成本节点 owned 图像：`TryCopyBorrowed(...)`
- 接管 HALCON 算子返回的 owned 对象：`CopyOwnedObjectOrNull(...)`
- 释放 owned 对象并识别 4051：`TryDisposeOwned(...)`、`IsDeletedObjectError(...)`

也就是说，它解决的是“边界怎么复制、怎么释放、怎么判断对象已死”，不是自动替所有模块保管所有 `HObject`。它像仓库里的标签枪：对象谁借的、谁拥有、谁释放，还是要模块自己贴清楚标签。

## 3. 新做法一：线扫片材计数

所属模块与文件路径：`Tools/GeneralALGO/ALGO.LineScanSheetCounter/Models/LineScanSheetCounterModel.cs`

### 3.1 输入阶段

线扫现在的关键思路是：如果输入参数里已经是本节点缓存的 owned `HImage`，执行时可以只借用；如果是普通 `HObject`，先复制出 `HImage` 再进算法。

核心形态：

```csharp
// Tools/GeneralALGO/ALGO.LineScanSheetCounter/Models/LineScanSheetCounterModel.cs
if (!TryResolveProcessImage(inputObject, out HImage? input, out bool ownsInput, out string error))
{
    return NodeStatus.None;
}

try
{
    using LineScanSheetCounterResult result = _algorithm.Process(
        _runtimeRemainImage,
        input,
        _runtimeRemainMaskImage,
        mask,
        parameters,
        createPreviewImages: shouldCreatePreviewImages);
}
finally
{
    if (ownsInput)
    {
        HalconImageOwnership.DisposeOwned(input);
    }
}
```

要点：

- `InputImage.Value` 如果由本节点 `ResolveAndApplyInputImage()` 复制出来，就打上 `_ownsInputImageValue = true`。
- 执行阶段不要无脑 `CopyImage()`，否则线扫大图会像复印机卡纸，一帧复制几份，内存很快爆。
- mask 和 input 用同一套规则处理，避免一边 owned、一边 borrowed 造成残留图不同步。

### 3.2 算法阶段

所属模块与文件路径：`Tools/GeneralALGO/ALGO.LineScanSheetCounter/Models/LineScanSheetCounterAlgorithm.cs`

算法层的规则是：

- 调用方已经给了稳定输入，算法层只读输入，不再重复复制整帧。
- `HOperatorSet.CropRectangle1(...)`、`TileImagesOffset(...)` 等算子 `out HObject` 产物属于当前算法。
- 算法要把结果交给 Model，就走 `AdoptImageObject(...)` 或 `CopyImageObject(...)`。

核心形态：

```csharp
// Tools/GeneralALGO/ALGO.LineScanSheetCounter/Models/LineScanSheetCounterAlgorithm.cs
HOperatorSet.CropRectangle1(_concatImage, out HObject targetImage, row1, col1, row2, col2);
_counterResult.TargetImages.Add(AdoptImageObject(targetImage));

private static HImage AdoptImageObject(HObject image)
{
    HImage? copiedImage = HalconImageOwnership.CopyOwnedObjectOrNull(image);
    if (copiedImage == null)
    {
        throw new InvalidOperationException("Failed to copy owned HALCON image.");
    }

    return copiedImage;
}
```

这里的重点不是“复制越多越安全”，而是“复制一次并把原 owned 产物释放掉”。这叫接管，不叫养殖场无限繁殖。

### 3.3 结果交接阶段

所属模块与文件路径：`Tools/GeneralALGO/ALGO.LineScanSheetCounter/Models/LineScanSheetCounterResult.cs`

线扫结果类提供了 `DetachRemainImage()`、`DetachTargetImages()` 这类方法。这个设计很关键：

- `Result` 被 `using` 包起来，离开作用域会 Dispose。
- 如果 Model 要长期持有结果图，就必须先 Detach，把所有权从 Result 转移出来。
- Detach 后 Result Dispose 不会再释放这些对象，避免一边输出、一边被结果对象清理掉。

标准写法：

```csharp
// Tools/GeneralALGO/ALGO.LineScanSheetCounter/Models/LineScanSheetCounterModel.cs
_runtimeRemainImage = result.DetachSourceRemainImage() ?? result.DetachRemainImage();
_runtimeRemainMaskImage = result.DetachRemainMaskImage();
LastConcatImage = result.DetachLastConcatImage();
AdoptTargetImages(result.DetachTargetImages());
```

适合复用的规则：只要算法结果对象实现 `IDisposable`，并且里面带 `HImage/HObject`，就优先做 Detach，而不是让外层去猜哪些对象还能活。

## 4. 新做法二：区域处理

所属模块与文件路径：`Tools/GeneralALGO/ALGO.RegionProcessing/Models/RegionProcessingModel.cs`

区域处理比线扫多了一个特点：它不只是图像，还有 `HRegion/HObject` 区域输出、区域预览、步骤中间结果。因此它适合当“区域类节点”的标准样板。

### 4.1 输入图像接管

区域处理现在先解析链接值，再复制成本节点 owned 输入图：

```csharp
// Tools/GeneralALGO/ALGO.RegionProcessing/Models/RegionProcessingModel.cs
object imageValue = ResolveLinkedInputImageValue();
HImage ownedImage = CreateOwnedInputImage(imageValue);
if (!HalconImageOwnership.IsInitializedSafe(ownedImage))
{
    HalconImageOwnership.DisposeOwned(ownedImage);
    _inputImage.Value = null;
    _ownsInputImage = false;
    return;
}

_inputImage.Value = ownedImage;
_ownsInputImage = true;
```

关键点：

- 从上游链接、全局参数、缓存里来的图像默认 borrowed。
- 只有 `CreateOwnedInputImage(...)` 成功复制出来的图像，才允许 `_ownsInputImage = true`。
- 释放输入图时只释放 `_ownsInputImage == true` 的对象。

### 4.2 区域对象接管

区域对象不能只套用图像尺寸判断，所以区域处理保留了模块内 `CloneHObject(...)` / `CloneHRegion(...)`：

```csharp
// Tools/GeneralALGO/ALGO.RegionProcessing/Models/RegionProcessingModel.cs
private static HObject? CloneHObject(HObject? source)
{
    if (source == null || !HalconImageOwnership.IsInitializedSafe(source))
    {
        return null;
    }

    HOperatorSet.CopyObj(source, out HObject copied, 1, -1);
    return copied;
}
```

这块后续可以进一步沉淀进 `HalconImageOwnership`，例如补 `TryCopyBorrowedObject(...)`，但当前区域处理已经把规则收口到了少数几个函数里，风险比散落全文件直接 `Clone()` 小很多。

### 4.3 输出退休机制

区域处理的亮点是 `RetireOutputHObject(...)`：旧输出不一定立刻杀掉，而是放进一个小队列，超过数量再释放。

```csharp
// Tools/GeneralALGO/ALGO.RegionProcessing/Models/RegionProcessingModel.cs
private void RetireOutputHObject(HObject? hObject, string label)
{
    if (hObject == null)
    {
        return;
    }

    if (!CanRetainOutputHObject(hObject))
    {
        SafeDisposeOutputHObject(hObject, label);
        return;
    }

    _retiredOutputHObjects.Enqueue((hObject, label));
}
```

为什么要退休队列？因为输出对象可能刚写进 `OutputParams`，下游/UI/预览还在短时间内拿着引用。如果上一帧刚输出，下一帧马上释放旧对象，下游就可能拿到“死亡螺丝刀”，日志里就是 `#4051 object has been deleted already`。

### 4.4 释放策略

```csharp
// Tools/GeneralALGO/ALGO.RegionProcessing/Models/RegionProcessingModel.cs
private static void SafeDisposeOutputHObject(HObject? hObject, string label)
{
    if (hObject == null)
    {
        return;
    }

    if (HalconImageOwnership.TryDisposeOwned(hObject, out Exception disposeError))
    {
        return;
    }

    if (HalconImageOwnership.IsDeletedObjectError(disposeError))
    {
        return;
    }

    WriteStaticLog("ERROR", $"输出对象释放失败: Label={label}, Error={disposeError}");
}
```

标准口径：释放路径遇到 4051，不要把主流程打成 Error。它本质是“垃圾桶里已经没垃圾了，你又倒了一次”，没必要报警说垃圾桶爆炸。

## 5. 旧做法对比

旧模块不是全错，它们在小图、单节点、低频预览时能跑。但一进入连续线扫、大图、多下游、异步预览，就容易出问题。

| 对比点 | 旧做法 | 新标准做法 |
| --- | --- | --- |
| 输入复制 | `inputImage.Clone()`、`new HImage(inputObject).Clone()` 分散在各模块 | 统一走 `HalconImageOwnership.TryCopyBorrowed(...)` 或模块内安全包装 |
| 活性判断 | 直接 `IsInitialized()` | `IsInitializedSafe(...)`，图像再用 `TryGetImageSize(...)` 验证 |
| 释放 | 直接 `Dispose()`，异常吞掉或打断流程 | `DisposeOwned(...)` / `TryDisposeOwned(...)`，4051 统一识别 |
| 输出替换 | 直接覆盖字段或参数值 | 替换前释放/退休旧 owned 输出，必要时延迟释放 |
| 算法结果 | 返回对象后谁释放不够明确 | Detach 转移所有权，或 Copy 后释放 owned 原对象 |
| 跨 UI/异步 | 可能直接拿原对象显示 | 先复制给 UI，Dispatcher 回调 finally 释放 |

旧做法典型片段：

```csharp
// Tools/GeneralALGO/ALGO.CreatRegion/Models/CreatRegionModel.cs
case HImage inputImage:
    return inputImage.Clone();
case HObject inputObject:
    return new HImage(inputObject).Clone();
```

```csharp
// Tools/GeneralALGO/ALGO.RegionTrans/Models/RegionTransModel.cs
case HRegion region:
    return region.Clone();
case HObject hObject when hObject.IsInitialized():
    return hObject.Clone();
```

```csharp
// Tools/GeneralALGO/ALGO.DistanceLL/Models/DistanceLLModel.cs
if (_inputImage?.Value is HImage inputImage)
{
    inputImage.Dispose();
}
```

这些写法的问题：

- `new HImage(inputObject)` 很容易让人误以为“已经深拷贝”，实际它更像给同一把钥匙换了个钥匙扣。
- 直接 `IsInitialized()` 遇到底层对象已删时可能抛 HALCON 异常。
- 直接 `Dispose()` 不区分 borrowed / owned，一旦释放上游共享对象，下游就会炸 4051。
- 没有统一的输出退休策略，连续运行时更容易“上一帧对象被下一帧清掉”。

## 6. 标准化接管流程

### 6.1 第一步：先画所有权边界

新增或改造节点前，先把对象分成四类：

- Borrowed 输入：来自 `InputParams`、`NodesOutputCache`、全局参数、上游输出。
- Owned 输入缓存：本节点复制后保存在字段里的输入图/区域。
- Owned 算法临时对象：HALCON 算子 `out HObject` 返回的对象。
- Owned 输出对象：本节点写入 `OutputParams`、预览、运行态字段的对象。

没有贴标签的 `HObject` 最危险，就像车间里没挂牌的电源线，迟早有人拔错。

### 6.2 第二步：输入统一复制

```csharp
// 示例路径：Tools/GeneralALGO/<YourModule>/Models/<YourModel>.cs
private HImage? CreateOwnedInputImage(object? imageValue)
{
    if (imageValue is not HObject image)
    {
        return null;
    }

    return HalconImageOwnership.TryCopyBorrowed(image, out HImage ownedImage)
        ? ownedImage
        : null;
}
```

规则：

- 需要跨周期保存：复制。
- 需要跨线程预览：复制。
- 只在当前同步调用栈读取，且已经确认是本节点 owned 缓存：可以借用。
- 不确定：复制。工业现场宁可多拧一颗螺丝，也别让整线停机。

### 6.3 第三步：算法临时对象统一收口

```csharp
// 示例路径：Tools/GeneralALGO/<YourModule>/Models/<YourAlgorithm>.cs
HObject? tempRegion = null;
try
{
    HOperatorSet.Threshold(inputImage, out tempRegion, minGray, maxGray);
    return CloneHObject(tempRegion) ?? new HObject();
}
finally
{
    HalconImageOwnership.DisposeOwned(tempRegion);
}
```

如果是图像结果，并且 `tempImage` 是 HALCON 算子刚返回的 owned 对象：

```csharp
// 示例路径：Tools/GeneralALGO/<YourModule>/Models/<YourAlgorithm>.cs
HOperatorSet.CropRectangle1(inputImage, out HObject rawImage, row1, col1, row2, col2);
return HalconImageOwnership.CopyOwnedObjectOrNull(rawImage);
```

### 6.4 第四步：输出边界复制或 Detach

两种标准选一：

- 算法结果对象可控：做 `Detach...()`，把所有权从 Result 转给 Model。
- 算法结果不可控：写入 `OutputParams` 前复制一份稳定对象。

```csharp
// 示例路径：Tools/GeneralALGO/<YourModule>/Models/<YourModel>.cs
private void ReplaceOutputImage(HImage? nextImage)
{
    HImage? old = OutputImage;
    OutputImage = nextImage;

    if (!ReferenceEquals(old, nextImage))
    {
        HalconImageOwnership.DisposeOwned(old);
    }
}
```

如果该输出会被下游/UI 短时间引用，参考区域处理做退休队列，不要立即释放旧输出。

### 6.5 第五步：预览对象单独复制

```csharp
// 示例路径：Tools/GeneralALGO/<YourModule>/Models/<YourModel>.cs
HImage? previewCopy = HalconImageOwnership.CopyOwnedOrNull(sourceImage);
try
{
    Application.Current.Dispatcher.Invoke(() =>
    {
        mWindowH?.HobjectToHimage(previewCopy);
    });
}
finally
{
    HalconImageOwnership.DisposeOwned(previewCopy);
}
```

预览失败不应影响主流程。预览是仪表盘，不是发动机；仪表盘闪一下可以报警，不能把发动机当场拆了。

## 7. 其他模块改造时的选择建议

- 图像连续流、线扫、拼接、裁图类模块：优先照线扫片材计数。
- 区域生成、区域过滤、区域转换、多步骤区域处理类模块：优先照区域处理。
- 简单测量、低频标定类模块：可以先只替换输入复制、活性判断、释放方法，不必一次上完整退休队列。
- 老模块如果还在用 `new HImage(hObject).Clone()`、裸 `IsInitialized()`、裸 `Dispose()`，都列入候选收口清单。

优先改造顺序：

1. 连续运行且处理大图的节点。
2. 输出会被多个下游共享的节点。
3. 有 Dispatcher/UI 预览的节点。
4. 日志里出现过 `#4051`、`clear_obj`、`get_image_size` 的节点。

## 8. 遗留收口点

- `Application/NodifyFlow/Nodify.FlowApp/ViewModels/Node/NodeViewModel.cs` 已经有下游参数克隆和旧输入释放机制，但里面仍有裸 `IsInitialized()`、`CopyImage()`、`CopyObj()`、`Dispose()`；后续可以考虑也改成调用 `HalconImageOwnership`，让流程引擎边界和工具节点边界口径完全一致。
- `HalconImageOwnership` 当前偏图像对象，区域对象还主要由 `RegionProcessingModel.CloneHObject(...)` 自己处理；后续可补 `TryCopyBorrowedObject(...)` / `CopyOwnedObjectAndDispose(...)` 这类更通用的方法。
- `ALGO.CreatRegion`、`ALGO.RegionTrans`、`ALGO.DistanceLL` 是最直观的旧写法样本，后续如果它们进入连续流程，应优先收口。

## 9. 验收清单

改造一个模块后，至少看这些点：

- [ ] 输入来自上游时，没有直接释放 `InputParams` 里的对象。
- [ ] 图像有效性判断使用 `TryGetImageSize(...)`。
- [ ] 算子 `out HObject` 临时对象有明确释放路径。
- [ ] 写入 `OutputParams` 的对象是稳定副本或明确 Detach 后的 owned 对象。
- [ ] UI/Dispatcher 使用的是预览副本，不是运行态原对象。
- [ ] 日志不再出现 `HALCON error #4051`、`object has been deleted already`。
- [ ] 连续运行时 `PrivateMB` 不再阶梯式无回落上涨。
- [ ] 旧 DLL 已覆盖到 `OutputExe/Debug/net8.0-windows/Modules`。

## 10. 后续转 Skill 的骨架

后面要做成 Codex Skill 时，可以按这个结构拆：

- 触发条件：用户提到 HALCON、HObject、HImage、HRegion、4051、get_image_size、clear_obj、对象释放、图像内存上涨。
- 第一步：先找 2 到 3 个仓库对照模块，优先线扫、区域处理、深度学习。
- 第二步：画 borrowed / owned / temporary / output 四类对象边界。
- 第三步：按输入、算法、输出、预览、释放五段改造。
- 第四步：构建模块并复制 DLL 到 `OutputExe/Debug/net8.0-windows/Modules`。
- 第五步：查日志验证 `#4051`、节点状态、内存曲线。

一句话：HALCON 对象不是 C# 普通对象，它更像工厂里的夹具。谁借、谁复制、谁归还必须写清楚；不然夹具还没回库，下一个工位已经拿着空气开干了。
