# Core 领域规则

本文件继承仓库根级 [AGENTS.md](../AGENTS.md)，只能收紧或细化根级门禁，不得放宽安全、验证、评审、授权或例外要求。规则冲突时执行更严格者并停止冲突范围内的修改，交由负责人裁决。

## 适用范围

适用于 `Core/` 及其全部子目录中的源码、项目文件、配置、数据库/缓存基础设施、`Core/ReeYin_V.UI` 共享 UI、测试、文档和生成规则。更深层 `AGENTS.md` 可以继续收紧本文件。

## 修改前调查与阅读

先按 [变更影响地图](../docs/development/change-impact-map.md) 定位 Core 消费者，再从 [构建与测试地图](../docs/development/build-and-test-map.md) 选择验证入口。阅读范围按 [开发规范索引](../docs/development/README.md#which-document-to-read按变更选择文档) 和实际风险确定；Lite 只可排除无关专题，不能省略公共契约、数据、线程或生命周期调查。涉及架构/公共契约、实现、验证、评审、AI、模块生命周期或安全时，分别读取对应专题。

## 公共契约与依赖方向

- `Core` 只拥有基础设施、共享抽象、跨模块契约、配置/缓存/数据库基座、公共服务和事件定义，以及由 `Core/ReeYin_V.UI` 提供的共享控件、主题、样式和资源字典。禁止新增对 `Shell`、具体 `Application`、`Tools`、硬件厂商实现、`CustomizedDemand` 或客户类型的依赖，也禁止用 DLL 文件引用、反射、复制类型或运行时服务定位绕过分层。
- 仓库中已存在的反向引用只是 legacy 基线，不是新增依赖或职责错位的先例。修改含 legacy edge 的项目时，必须列出精确 `ProjectReference`、实际类型/成员/命名空间 usage、此次是否扩大以及目标迁移方向；未触及的 edge 不在当前任务中顺手清理。
- 新增或沿现有反向 edge 扩大引用、usage 或错层职责默认禁止。只有任务明确要求，并在实施前完成可评审设计、兼容分析、失败/恢复与回滚方案、熟悉架构和领域的人类评审，以及根级例外记录后，才可在批准的 edge、usage、职责、期限和范围内实施。AI 和作者不得自行批准；例外不得成为其他依赖的先例。
- 公开类型、接口、DTO、事件负载、程序集/类型标识、默认行为、异常/取消/超时语义、服务键和模块标识均是消费者契约。变更时至少按 R2 处理，定位可找到的消费者并由熟悉该契约的人类评审；无法定位全部消费者时必须披露搜索范围和未知风险。

## 共享 UI 权威所有权

- 当前共享控件、主题、样式、资源字典及其公共资源键的权威所有者是 `Core/ReeYin_V.UI`。修改这些契约时必须在 Core 范围内定位 XAML、控件、主题和模块消费者，验证资源 URI、合并顺序、默认样式、亮/暗/定制主题和运行时加载兼容。
- `CustomUI/` 当前是项目、业务或客户组合 UI 及扩展区，必须复用 `Core/ReeYin_V.UI` 的共享资源；目录名称不表示共享 UI 已迁移，也不授权在 `CustomUI` 重定义全局资源键或复制共享模板形成第二权威来源。
- 将共享 UI 所有权迁移到 `CustomUI` 必须作为独立设计和至少 R2 的公共契约变更，明确资源/程序集身份、全部可定位消费者、兼容窗口、迁移顺序、运行时加载验证和回滚。迁移完成并验证前，`Core/ReeYin_V.UI` 的现有路径和所有权保持有效。

## 数据、缓存与序列化兼容

- 必须保持序列化类型/成员名、类型、默认值、枚举数值和忽略/包含属性，以及 Recipe 名称/路径/应用时机、Output 名称/类型/可写性/刷新时机和缓存键格式兼容。变更必须验证旧数据读取、新数据往返、缺失/额外/错误字段和失败恢复。
- 缓存和全局注册表必须定义键命名空间、大小写、生命周期、并发策略、失效条件和唯一所有者。节点缓存沿用 `{serial:D3}`、`TryGetValue` 和类型检查；改键必须提供显式迁移、兼容窗口和回滚，禁止让两个无迁移关系的键格式长期并存。
- 数据库表、列、约束、默认值、连接/加密配置和事务边界不得静默变化。迁移必须可重复、有版本识别，在非生产脱敏数据上验证升级、事务失败、旧数据读取、备份/恢复和回滚；生产数据读取或变更必须另行取得限定数据目标、操作、执行人、窗口、检查及停止/回滚/恢复的生产数据变更授权，AI 不得修改生产数据。
- 迁移属于至少 R3 的受控变更；若可能造成设备、人身或生产数据不可逆风险则为 R4。R3/R4 的领域评审、变更负责人批准、停止/回滚方案和风险披露不可豁免；R4 还需具体执行批准，且不得由 AI 执行。

## 全局事件、线程与生命周期

- 新增或修改全局事件前必须明确事件定义所有者、发布者、订阅者、负载契约、线程上下文、顺序假设、异常边界和取消订阅责任。发布方不得依赖订阅顺序，负载不得携带 WPF 控件、厂商句柄或所有权不明的可释放资源。
- 全局 `EventAggregator`、静态 provider 或 DI 解析只提供访问，不转移资源所有权。短生命周期对象订阅长生命周期发布者时必须保存命名委托或订阅令牌并可靠解绑；重复初始化、反序列化和导航往返不得累积订阅。
- 共享缓存、注册表、事件和后台任务必须有明确同步策略。禁止在持锁区调用不可控外部代码或阻塞异步工作；UI 状态只在 Dispatcher/UI 线程更新，耗时、数据库、文件和 Native 工作不得占用 UI 线程。
- 初始化必须可重复判断并处理部分失败；只有必需状态已建立后才能发布就绪。`Dispose`/`DisposeAsync` 必须幂等，覆盖失败、取消、超时、模块卸载、方案切换和退出，先停止自有任务并解绑/释放自有资源，再执行框架要求的基类清理；借用资源不得擅自释放。

## 修改该目录必须完成

以下清单按实际影响和 R0-R4 执行；不适用项必须记录原因，不能虚构为通过。纯 R0 文档变更执行 L0 差异、内容、路径和链接检查；R3/R4 的领域/安全评审、变更负责人批准、停止/回滚、风险披露及 R4 具体执行批准仍按根级强门禁执行。

- [ ] 写明问题、成功标准、非目标、最高风险等级、资源所有者和完整调用/消费者链。
- [ ] 审计项目引用及代码 usage；含 legacy edge 时记录 edge、是否扩大、迁移方向和所需受控例外。
- [ ] 逐项给出公共 API、事件、配置、序列化、Recipe、Output、缓存、数据库和 `Core/ReeYin_V.UI` 共享资源兼容结论。
- [ ] 覆盖并发、线程、重复初始化、部分失败、取消、超时、停止和幂等释放路径。
- [ ] 从最终差异和静态检查开始，构建最小受影响项目、相关测试及可定位消费者；按风险补充兼容、迁移、恢复和模块加载场景。
- [ ] 数据迁移在非生产脱敏数据上验证，并提供可执行回滚、备份/恢复前提和失败后的安全停止路径。
- [ ] 记录最终修改后的命令/场景证据、失败/未验证项、残余风险和所需人类评审/批准；提交、发布、生产数据和 R4 操作仍需各自独立授权。
