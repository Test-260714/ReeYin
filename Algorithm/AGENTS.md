## 输入输出与数值契约

- 每个公共或 Native 导出入口必须定义输入/输出形状、元素类型、通道/步长、字节序、单位、坐标系、轴顺序、原点、方向、范围、返回码和错误所有权；禁止由调用方根据数组位置、历史默认值或实现细节猜测。
- 必须显式处理 `null`/空输入、零尺寸、非有限值、退化几何、非法枚举、维度/长度不一致、超大图像/点云/批次，以及尺寸乘法、索引和字节数溢出。失败时输出应保持契约定义的空/未修改状态，不得暴露部分初始化缓冲区。
- 浮点、几何、标定和拟合比较必须根据领域定义绝对/相对容差、单位和边界行为，不能用机器 `epsilon` 代替业务容差。变更前记录基准结果与误差阈值；阈值变化属于算法契约变化，需要说明依据并验证消费者。
- 要求确定性的路径必须固定随机种子、输入顺序、并行归约策略和后端配置，并说明允许的跨 CPU/GPU/库版本差异。不具备确定性时必须明确输出波动范围和比较方法，禁止挑选单次有利结果作为证据。

## 性能、取消与资源所有权

- 性能改动必须先建立可复查基线，固定数据集类别/规模、预热、迭代次数、构建配置、CPU/GPU、线程数、依赖版本和计时范围；同时记录延迟/吞吐、峰值内存及正确性。优化不得以放宽精度、安全检查、取消或错误处理为代价。
- 长循环、批处理、模型推理和可能无限等待的外部库调用必须设计取消/超时边界。取消必须可观察并及时终止后续工作；只有在契约明确且调用方可识别时才能返回部分结果，取消和超时不得报告为成功。
- Native 句柄、指针、图像/点云缓冲和导出数组必须说明创建者、借用/接管语义、有效期、线程限制和唯一释放函数。跨 DLL 分配的内存必须由同一分配器配对释放；每个 `create`/分配出口必须有对应 `destroy`/`free`，且重复释放和 use-after-free 由契约与包装层防止。
- 托管互操作层必须使用与所有权匹配的 `SafeHandle`/幂等 `Dispose` 或等价封装，覆盖 P/Invoke 失败、部分初始化、异常、取消和重复关闭。固定内存、回调委托和借用指针的存活期必须覆盖 Native 调用，禁止依赖 GC 及时释放资源。

## 第三方依赖与结果证据

- 新增、替换或升级 OpenCV、PCL、VTK、推理后端、加密/压缩库及其他第三方组件必须单独审查来源、精确版本、许可证、安全、CPU/GPU 与运行时要求、目标架构、API/ABI、模型/驱动兼容、发布复制和回滚版本。
- 公共导出符号、结构体布局、调用约定、字符编码、布尔/枚举宽度和内存释放函数都是 ABI 契约。修改时必须构建对应架构和可定位的托管/Native 消费者，并验证旧调用方式或提供经批准的迁移窗口。
- 算法结果证据必须使用来源明确且脱敏的固定数据集，记录样例类别、期望来源、指标、容差、实际结果、失败样例和环境；覆盖正常、边界、非法、退化、超大输入、重复调用和并发。视觉抽查只能补充量化/黄金结果，不能替代它们。
- 算法输出可能驱动运动、判定、报警或生产数据时，必须沿消费者链提升到对应 R3/R4 门禁。真实设备操作授权和生产数据变更授权必须分别限定对象、操作、执行人、窗口、检查及停止/回滚/恢复；R4 还需具体执行批准，AI 不得执行。

## 修改该目录必须完成

以下清单按实际影响和 R0-R4 执行；不适用项必须记录原因，不能虚构为通过。纯 R0 文档变更执行 L0 差异、内容、路径和链接检查；R3/R4 的领域/安全评审、变更负责人批准、停止/回滚、风险披露及 R4 具体执行批准仍按根级强门禁执行。

- [ ] 涉及算法输入输出或数值行为时，写明算法契约、单位/坐标、数据规模、容差、确定性要求、资源所有者、成功标准、非目标和最高风险等级。
- [ ] 覆盖 `null`、空、边界、非有限、退化、超大输入、溢出、取消/超时、重复调用、并发和失败清理。
- [ ] 审计所有 Native 创建/销毁、分配/释放、回调和托管 `Dispose` 配对，验证部分初始化失败和重复释放路径。
- [ ] 审查第三方来源、版本、许可证、安全、API/ABI、架构、运行时复制与可执行回滚版本。
- [ ] 构建修改的 Native/托管项目及可定位消费者；按实际支持的配置/架构验证导出 ABI、加载和异常路径。
- [ ] 仅在精度、确定性或性能行为相关变更时，用最终版本在固定脱敏数据集上记录对应指标、容差或性能基线对比，并保留失败、跳过和未验证项。
- [ ] 交付兼容结论、回滚步骤、残余风险和所需人类领域评审/批准；涉及真实设备、生产数据或 R4 对象时保持独立授权边界。
