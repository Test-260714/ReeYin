# 贡献 ReeYin-V

本流程同时适用于人类开发者和 AI。开始前必须阅读 [根级规则](AGENTS.md) 及目标目录的就近 `AGENTS.md`；子目录规则只能增加约束。详细检查表和专题标准见 [开发规范索引](docs/development/README.md)。

## 1. 问题定义

开始修改前必须写清：当前行为、期望行为、影响对象、成功标准、非目标和已知约束。功能需求应当包含正常路径、失败/取消/超时/恢复路径和兼容性要求。

Bug 的最小复现记录必须包含：

- 版本、分支或构建标识，以及操作系统、运行时、硬件/模拟环境；
- 前置条件、最小输入/配置/数据（敏感信息必须脱敏）；
- 可重复的操作步骤、发生频率和最小复现范围；
- 期望结果与实际结果；
- 异常、日志、截图或测试失败等原始证据及采集时间；
- 首次出现版本或最后已知正常版本（未知时明确写“未知”）；
- 安全、生产数据和用户影响，以及临时规避方式。

无法复现时不得猜测性修复；必须说明已尝试条件、证据缺口和验证假设的方法。

## 2. 本地调查

修改前必须检查工作区并保留他人改动，然后调查目标入口、完整调用链、公共消费者、配置/序列化/Recipe/Output/缓存/数据库、异步线程、异常处理和资源释放。必须查看同项目及相邻文件的局部模式、相关测试和实际可用的构建命令。

调查结论至少记录根因或实现假设、受影响文件、兼容边界和待验证风险。禁止仅凭文件名、单个搜索结果或历史日志下结论。

## 3. 风险分类

采用“取最高等级”原则；不确定时必须按较高等级处理。

| 等级 | 典型范围 | 最低门禁概览 |
| --- | --- | --- |
| R0 | 文档、注释且不改变运行行为 | 差异、内容、路径和链接检查 |
| R1 | 局部 UI、内部逻辑、非公共重构 | R0 + 修改项目构建 + 相关测试 |
| R2 | 公共 API、MVVM 生命周期、Recipe、Output、序列化、缓存、数据库 | R1 + 消费者构建 + 兼容性与场景测试；涉及公共契约/兼容边界时必须由熟悉该领域的人评审 |
| R3 | 硬件运动、报警、权限、安全、迁移、发布配置 | R2 + 失败/恢复路径 + 回滚方案 + 领域评审 |
| R4 | 对设备、人身或生产数据存在不可逆风险 | R3 + 真实环境验证方案 + 明确负责人批准 |

风险等级在实现中升高时必须暂停，重新确定设计、验证、评审和授权。R3/R4 禁止豁免安全评审、回滚/停止方案、负责人批准和风险披露，R4 操作不得交给 AI 执行。紧急情况下只可以缩短或延期非安全验证，并必须记录原因、替代验证、批准人、批准有效期限和补验时限；逾期未补验时必须撤销例外或停止交付/运行。

## 4. 必要设计

出现以下任一触发器时，编码前必须形成可评审设计：

- 新模块、跨项目/跨层职责变化或依赖方向变化；
- 公共 API、数据模型、数据库、序列化、Recipe、Output、缓存或动态加载契约变化；
- MVVM/节点生命周期、并发、线程、取消、重试或资源所有权变化；
- 硬件、运动、报警、互锁、权限、安全、网络、外部进程或生产数据变化；
- 破坏兼容、需要迁移/特性开关/分阶段发布，或缺少安全回滚；
- 有多个可行方案且取舍会影响行为、测试或运维。

设计必须说明目标与非目标、方案及取舍、组件职责和调用/数据流、兼容与迁移、失败与恢复、安全控制、验证矩阵、上线限制和回滚。评审通过前禁止实施受触发器约束的行为变更。

## 5. 范围内实现

- 只修改满足成功标准所需的最小范围；禁止夹带无关重构、批量格式化、依赖升级、生成物或旁支修复。
- 遵循就近模式和详细编码标准；旧模式与安全或契约冲突时，记录差异并按设计处理。
- 保留现有公共契约和数据可读性；必须改变时提供迁移、兼容窗口、消费者更新与回滚策略。
- 实现正常路径时同步实现输入校验、失败、取消、超时、停止、恢复和资源释放。
- 发现用户改动重叠或任务外缺陷时停止重叠修改并报告，禁止覆盖或擅自扩大范围。

## 6. 分层验证

验证必须在最终修改后运行并保留新鲜证据，按低成本到高风险逐层推进：

1. 检查差异、格式、链接、静态规则和敏感信息；
2. 构建最小受影响项目并运行直接相关测试；
3. 构建消费者，验证公共契约、配置、序列化、数据和模块加载兼容；
4. 执行正常、失败、取消、超时、停止、恢复和回滚场景；
5. 仅在明确授权和安全前置条件满足后执行集成、人工 UI 或真实设备验证。

证据必须按类型记录，禁止用一种证据冒充另一种：

- **命令证据**：精确命令、执行时间、环境、退出码、错误/失败/跳过计数和关键输出。工具不提供某项时必须明确写“不适用”，禁止猜测或编造数值。
- **人工场景证据**：操作步骤、环境、预期结果、实际结果、执行人、执行时间，以及适用时的截图或日志。人工步骤没有退出码或计数时必须写“不适用”，禁止补造命令式结果。

历史结果、修改前结果、模拟器结果和编译成功不得扩大解释范围。缺少依赖、许可证、硬件或权限时必须标记“未验证”，说明影响，不得写成通过。既有失败必须与本次回归区分并附证据，禁止隐藏或删除失败测试。

## 7. 自审

提交评审前必须逐项检查：差异是否只包含任务范围；根因/成功标准是否已覆盖；调用方与兼容路径是否完整；异常、并发、取消、释放和安全路径是否可靠；日志是否可诊断且无敏感信息；测试是否可能误通过；文档、迁移和回滚是否与实现一致。

自审发现的问题必须修正并重新运行受影响验证。无本次验证证据不得声称完成。

## 8. 领域评审

R2 涉及公共契约或兼容边界时，必须由熟悉该领域的人评审；R3 必须获得硬件/安全/数据/发布等对应领域评审；R4 必须由明确负责人批准真实环境方案和操作窗口。作者不得批准自己的例外。

评审必须基于问题定义、风险等级、设计、完整差异、验证证据、未验证项、兼容结论和回滚方案。意见解决后若行为或风险发生变化，必须重新验证并请求相关部分复审。紧急修复也禁止省略安全评审、回滚/停止方案、负责人批准或风险披露；仅非安全验证可以按第 3 节的记录和限期要求缩短或延期。

## 9. 交付

交付说明必须包含：

- 修改内容、原因、文件/组件和明确非目标；
- 风险等级及其理由；
- 新鲜验证证据和结果；
- 未验证/失败项、原因、影响与跟进责任；
- API、配置、数据、Recipe、Output、缓存、UI 和设备行为的兼容结论；
- 回滚步骤、触发条件、备份/恢复要求及残余风险；
- 所需评审、批准和运行限制。

提交、推送、合并、发布、部署、生产数据变更和真实设备操作均需要用户或负责人分别明确授权；完成实现不等于获得这些授权。AI 禁止自行执行或批准上述动作，也禁止批准任何规则例外。
